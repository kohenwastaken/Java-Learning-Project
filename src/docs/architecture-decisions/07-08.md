# Architecture Decisions

## BankService Refactor, Result Enum ve Encapsulation Kararları

### Karar 1: CLI kodu ile business logic ayrıldı

`Main` classının sadece kullanıcı arayüzü gibi davranmasına karar verildi.

`Main` şunları yapar:

- Menü gösterir
- Input alır
- Service methodlarını çağırır
- Dönen sonuca göre kullanıcıya mesaj yazar

`BankService` şunları yapar:

- Customer, Account ve Transaction listelerini yönetir
- Register ve login işlemlerini yapar
- Para yatırma, çekme ve transfer işlemlerini yönetir
- Başarılı işlemlerde transaction log oluşturur
- Hesaba ait transactionları filtreler

Bu kararın sebebi, ileride CLI yerine REST API veya başka bir arayüz geldiğinde business logic'in tekrar kullanılabilir olmasıdır.

---

### Karar 2: BankService içinde Scanner veya IO.println kullanılmayacak

`BankService` içinde kullanıcıdan input alınmayacak ve kullanıcıya doğrudan çıktı yazdırılmayacak.

Kötü örnek:

```java
IO.println("Yetersiz bakiye");
```

Service içinde bunun yerine result döndürülür:

```java
return WithdrawResult.INSUFFICIENT_BALANCE;
```

Mesajı `Main` basar.

Bu karar, service layer'ın UI'dan bağımsız kalması için alındı.

---

### Karar 3: BankService aktif kullanıcı tutmayacak

`BankService` içinde `loggedAccount` veya `loggedCustomer` gibi bir field tutulmamasına karar verildi.

Bunun yerine:

```java
Customer customer = login(sc, bankService);
accountMenu(sc, bankService, customer);
```

Main login olmuş customer'ı bilir. İşlem yapılırken ilgili `accID` service'e gönderilir.

Örnek:

```java
bankService.depositToAccount(customer.getAccID(), depositAmount);
```

Bu karar, service'in state yönetimini azaltmak ve ileride Spring Boot yapısına daha uygun olmak için alındı.

---

### Karar 4: Account arama iki farklı methodla yönetilecek

İki farklı hesap arama durumu vardır.

#### 1. Sistemin kesin bulması gereken hesap

Login olmuş kullanıcının hesabı bulunmalıdır. Bulunamıyorsa bu sistem hatasıdır.

Bu yüzden:

```java
findAccountByID(int userID)
```

hesap bulamazsa exception fırlatır.

#### 2. Kullanıcının input olarak verdiği hedef hesap

Transferde hedef ID yanlış girilebilir. Bu beklenen kullanıcı hatasıdır.

Bu yüzden:

```java
searchAccountByID(int targetID)
```

hesap bulamazsa `null` döndürür.

Bu ayrım şu prensibe göre yapıldı:
```
Beklenmeyen sistem hatası -> exception
Beklenen kullanıcı hatası -> result enum
```

---

### Karar 5: İşlem sonuçları için enum kullanılacak

Para yatırma, para çekme ve transfer işlemlerinde sadece `boolean` dönmek yetersiz kalabilir.

Bu yüzden şu enumlar oluşturuldu:

`DepositResult`
`WithdrawResult`
`TransferResult`

Bu kararın sebebi, `Main` tarafında kullanıcıya daha net mesaj verebilmektir.

Örneğin `WithdrawResult`:

```java
public enum WithdrawResult {
SUCCESS,
INVALID_AMOUNT,
INSUFFICIENT_BALANCE
}
```

Bu sayede false sonucunun ne anlama geldiğini tahmin etmek zorunda kalmıyoruz.

---

### Karar 6: Enumlar BankService içinde değil, ayrı dosyalarda tutulacak

Enumlar `BankService` içinde tutulursa `Account` classı `BankService`e bağımlı hale gelir.

Bu istenmeyen bağımlılık:
```
Account -> BankService.WithdrawResult
```

Bu yüzden enumlar ayrı dosyalara taşındı.

Yeni yapı:

```
Account -> WithdrawResult
BankService -> WithdrawResult
Main -> WithdrawResult
```

Bu karar, domain classlarının service classına bağımlı olmaması için alındı.

### Karar 7: Fieldlar private yapılacak

`Account`, `Customer`, `Transaction` ve `BankService` içindeki fieldların doğrudan dışarıdan erişilebilir olmaması gerektiğine karar verildi.

Örneğin `Account.balance` public olursa, başka bir class bakiyeyi kuralsız şekilde değiştirebilir.

Bu yüzden:

```java
private BigDecimal balance;
```

kullanılır.

Bakiye sadece kontrollü methodlarla değişmelidir:

```java
moneyDeposit(amount)
moneyWithdraw(amount)
moneySend(amount)
moneyReceive(amount)
```

Bu kararın sebebi encapsulation'dır.

---

### Karar 8: Değişmemesi gereken ID değerleri final olabilir

`Account.accID`, `Customer.accID`, `Transaction.transactionID`, `sourceID`, `targetID`, `amount`, `type` gibi değerler transaction veya entity oluşturulduktan sonra değişmemelidir.

Bu yüzden bu tarz fieldlar için `final` kullanmak mantıklıdır.

Örnek:

```java
private final int accID;
```

Bu karar, sistemde ID tutarlılığını korumak için alındı.

---

### Karar 9: Transaction listesi doğrudan dışarı açılmayacak

`BankService` içindeki bütün transaction listesini doğrudan döndürmek doğru değildir.

Kötü fikir:

```java
getTransactionList()
```

Çünkü dışarıdan liste değiştirilebilir veya service'in iç yapısı dışarı sızmış olur.

Bunun yerine:

```java
getTransactionsForAccount(int userID)
```

methodu yazıldı.

Bu method sadece ilgili hesaba ait transactionları filtreleyip yeni bir liste olarak döndürür.

Bu kararın sebebi, service iç state'ini korumak ve `Main` classının sadece ihtiyacı olan veriyi almasını sağlamaktır.

---

### Karar 10: Transaction log yazdırma Main tarafında kalacak

Transactionları filtreleme işi `BankService` tarafında yapılır.

Yazdırma işi ise `Main` tarafında kalır.

```text
BankService -> transactionları filtreler
Main -> transactionları ekrana basar
```

Bu karar, service layer'ın UI bağımsız kalması için alındı.

---

## Bugünkü mimari sonuç

Bugünün sonunda yapı şu hale geldi:

### Main
- CLI
- Scanner
- Menü
- Kullanıcı mesajları
- Result switchleri

### BankService
- Application service
- Register/login
- Deposit/withdraw/transfer flow
- Account lookup
- Transaction oluşturma
- Transaction filtreleme

### Account
- Balance kuralları
- Deposit/withdraw/send/receive işlemleri

### Customer
- Kullanıcı bilgileri
- Password match kontrolü

### Transaction
- Immutable işlem kaydı

### Result Enums
- DepositResult
- WithdrawResult
- TransferResult

Bu yapı, ileride repository, unit test, Spring Boot ve hexagonal architecture'a geçiş için daha uygun hale geldi.
