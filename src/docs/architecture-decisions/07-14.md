# Architecture Decisions

## 1. Veri erişimi repository yapısı üzerinden gerçekleştirilecek

### Karar

BankService; Account, Customer ve Transaction verilerine doğrudan `List`, `Map` veya başka bir koleksiyon üzerinden erişmeyecektir.

Veri erişimi şu repository interface’leri üzerinden yapılacaktır:

```text
AccountRepository
CustomerRepository
TransactionRepository
```

### Gerekçe

BankService’in sorumluluğu bankacılık iş akışlarını yürütmektir. Verinin nerede ve nasıl saklandığı BankService’in sorumluluğu değildir.

### Sonuç

BankService yalnızca repository sözleşmelerine bağımlı olacaktır. Saklama teknolojisi değiştiğinde iş mantığının değiştirilmesi gerekmeyecektir.

---

## 2. Repository interface’leri application outbound port olarak konumlandırılacak

### Karar

Repository interface’leri şu package altında tutulacaktır:

```text
org.example.bank.application.port.out
```

### Gerekçe

Bu interface’ler application katmanının dış dünyadan ihtiyaç duyduğu veri erişim işlemlerini tanımlar.

Application katmanı neye ihtiyaç duyduğunu belirtir, fakat işlemin nasıl gerçekleştirileceğini bilmez.

### Sonuç

Repository interface’i sözleşmeyi, outbound adapter ise bu sözleşmenin gerçek implementationını sağlayacaktır.

---

## 3. İlk repository adapterları in-memory olarak geliştirilecek

### Karar

İlk repository implementationları şu package altında bulunacaktır:

```text
org.example.bank.adapter.out.memory
```

Kullanılacak classlar:

```text
InMemoryAccountRepository
InMemoryCustomerRepository
InMemoryTransactionRepository
```

### Gerekçe

PostgreSQL’e geçmeden önce repository mimarisinin daha basit bir ortamda doğrulanması amaçlanmıştır.

In-memory adapterlar ek kurulum gerektirmez ve unit testlerde kolayca kullanılabilir.

### Sonuç

Veriler şimdilik uygulama belleğinde tutulacak ve uygulama kapandığında silinecektir.

---

## 4. Account ve Customer için Map, Transaction için List kullanılacak

### Karar

Account ve Customer verileri ID ile tekil olarak arandığı için `Map` kullanılacaktır:

```java
Map<Integer, Account>
Map<Integer, Customer>
```

Transaction verileri için başlangıçta `List` kullanılacaktır:

```java
List<Transaction>
```

### Gerekçe

Account ve Customer için erişim şekli şöyledir:

```text
ID → Tek nesne
```

Bir hesabın ise birden fazla transaction kaydı olabilir. Bu nedenle transactionlar liste içinde tutulup account ID’ye göre filtrelenecektir.

### Sonuç

Mevcut proje ölçeğinde basit ve okunabilir veri yapıları kullanılacaktır. Veri miktarı büyüdüğünde daha gelişmiş indeksleme yöntemleri değerlendirilebilir.

---

## 5. Bulunamayan tekil veriler Optional ile temsil edilecek

### Karar

Tek bir nesne döndüren repository arama methodları `Optional` kullanacaktır:

```java
Optional<Account> findById(int accountId);
Optional<Customer> findById(int customerId);
```

Liste döndüren methodlar ise veri bulunmadığında boş liste döndürecektir:

```java
List<Transaction> findByAccountId(int accountId);
```

### Gerekçe

`Optional`, bir nesnenin bulunamama ihtimalini method imzasında açıkça gösterir.

Collection döndüren methodlarda boş liste kullanılması, gereksiz `null` kontrollerini engeller.

### Sonuç

Repository sonuçlarında bulunamama durumları daha güvenli ve anlaşılır şekilde ele alınacaktır.

---

## 6. BankService constructor dependency injection kullanacak

### Karar

BankService’in ihtiyaç duyduğu repository’ler constructor üzerinden dışarıdan verilecektir:

```java
public BankService(
        CustomerRepository customerRepository,
        AccountRepository accountRepository,
        TransactionRepository transactionRepository
)
```

### Gerekçe

BankService’in kendi içinde `InMemoryAccountRepository` gibi somut classlar oluşturması, iş mantığını saklama teknolojisine bağımlı hâle getirirdi.

Constructor injection sayesinde BankService yalnızca repository interface’lerini bilir.

### Sonuç

BankService aynı kodla farklı repository implementationları kullanabilecektir:

```text
InMemoryAccountRepository
PostgreSqlAccountRepository
TestAccountRepository
```

---

## 7. Dependency bağlantıları şimdilik Main ve testler içinde kurulacak

### Karar

Spring Boot kullanılmadan önce repository nesneleri `Main` içinde oluşturulup BankService’e verilecektir.

```java
BankService bankService = new BankService(
        new InMemoryCustomerRepository(),
        new InMemoryAccountRepository(),
        new InMemoryTransactionRepository()
);
```

Unit testlerde de her testten önce temiz repository nesneleri oluşturulacaktır:

```java
@BeforeEach
void setUp() {
    bankService = new BankService(
            new InMemoryCustomerRepository(),
            new InMemoryAccountRepository(),
            new InMemoryTransactionRepository()
    );
}
```

### Gerekçe

Şu anda nesneleri otomatik oluşturup birbirine bağlayan bir Spring containerı bulunmamaktadır.

Testlerde her test için yeni repository oluşturulması, testlerin birbirini etkilemesini engeller.

### Sonuç

`Main` geçici olarak composition root görevi görecektir. Spring Boot aşamasında dependency oluşturma ve injection işlemleri Spring tarafından yönetilecektir.

---

## 8. BankService yalnızca iş akışlarını yönetecek

### Karar

BankService içindeki veri koleksiyonları ve veri arama döngüleri kaldırılacaktır.

Repository adapterlarına taşınacak sorumluluklar:

- Veriyi saklamak
- ID ile veri bulmak
- Transaction listesini account ID’ye göre filtrelemek

BankService içinde kalacak sorumluluklar:

- Kullanıcı kaydı
- Login ve şifre kontrolü
- Para yatırma
- Para çekme
- Para transferi
- İşlem sonucuna karar verme

### Gerekçe

BankService’in hem iş kurallarını hem de veri saklama işlemlerini yönetmesi birden fazla sorumluluk üstlenmesine neden oluyordu.

### Sonuç

Katmanların görevleri daha net ayrılacak ve uygulama PostgreSQL ile Spring Boot geçişine daha uygun hâle gelecektir.
