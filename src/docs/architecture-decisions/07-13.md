## Hexagonal Package Yapısı ve Repository Abstraction

### Karar 1: Classlar sorumluluklarına göre package'lara ayrılacak

Bütün classların tek bir `org.example` package'ında tutulmamasına karar verildi.

Yeni ana package yapısı:

```text
domain
application
adapter
```

Bu kararın amacı:

- Classların sorumluluklarını görünür hale getirmek
- Katmanlar arasındaki bağımlılıkları kontrol etmek
- Hexagonal architecture geçişine hazırlanmak
- Spring Boot ve PostgreSQL entegrasyonunu kolaylaştırmak

---

### Karar 2: Domain modelleri altyapı detaylarından bağımsız kalacak

Aşağıdaki classlar domain model içinde tutulacaktır:

```text
Account
Customer
Transaction
```

Domain classları şu detayları bilmeyecektir:

```text
CLI
REST Controller
Spring Boot
PostgreSQL
Docker
JPA
```

Bu karar, bankacılık kurallarının framework ve veritabanı teknolojilerinden bağımsız kalması için alındı.

---

### Karar 3: BankService veri koleksiyonlarını doğrudan yönetmeyecek

Mevcut yapıda `BankService` şu koleksiyonları doğrudan tutuyordu:

```java
List<Customer>
List<Account>
List<Transaction>
```

Bu yapı `BankService`i verinin bellekte `ArrayList` ile tutulmasına bağımlı hale getiriyordu.

Yeni yapıda `BankService` şu interface'lere bağımlı olacaktır:

```java
CustomerRepository
AccountRepository
TransactionRepository
```

Bu kararın amacı, business logic ile persistence sorumluluğunu ayırmaktır.

---

### Karar 4: Repository interface'leri outbound port olarak kullanılacak

Repository interface'leri application katmanında, outbound port olarak konumlandırılacaktır.

```text
application.port.out
```

Bu interface'ler yalnızca uygulamanın ihtiyaç duyduğu işlemleri tanımlayacaktır.

Örnek:

```java
public interface AccountRepository {

    Account save(Account account);

    Optional<Account> findById(int accountId);
}
```

Repository interface'lerinde aşağıdaki altyapı detayları bulunmayacaktır:

```text
ArrayList
HashMap
SQL
PostgreSQL
```

---

### Karar 5: İlk persistence adapterı in-memory olacak

Repository interface'lerinin ilk implementationları bellekte çalışacaktır.

Örnek:

```java
InMemoryAccountRepository
InMemoryCustomerRepository
InMemoryTransactionRepository
```

Bu classlar şu package altında tutulacaktır:

```java
adapter.out.memory
```

Bu kararın nedenleri:

- Repository pattern'i PostgreSQL eklemeden öğrenmek
- Mevcut CLI davranışını korumak
- Refactor sırasında aynı anda çok fazla teknoloji değiştirmemek
- Unit testleri hızlı ve bağımsız tutmak

---

### Karar 6: Account ve Customer repository'lerinde Map kullanılacak

`Account` ve `Customer` nesneleri çoğunlukla ID ile arandığı için in-memory implementationlarda `Map` kullanılmasına karar verildi.

Örnek:

```java
Map<Integer, Account>
Map<Integer, Customer>
```

Eşleşme:

```text
ID -> Domain nesnesi
```

Bu karar, ID ile yapılan aramaları listeyi tamamen dolaşmadan gerçekleştirmek için alındı.

Transaction kayıtlarında ise sorgulama ihtiyacına göre `List<Transaction>` kullanılabilir.

---

### Karar 7: findById sonucu Optional döndürecek

Repository `findById()` methodlarının nesne bulunamama ihtimalini açıkça ifade etmesi için `Optional` döndürmesine karar verildi.

Örnek:

```java
Optional<Account> findById(int accountId);
```

Bu kararın amacı:

- Kontrolsüz null kullanımını azaltmak
- Method sözleşmesini daha açık hale getirmek
- Hesap bulunamama durumunu çağıran katmana açıkça bildirmek

---

### Karar 8: BankService repository'leri constructor üzerinden alacak

`BankService`, repository implementationlarını kendi içinde oluşturmayacaktır.

Bunun yerine repository interface'leri constructor üzerinden verilecektir.

```java
public BankService(
CustomerRepository customerRepository,
AccountRepository accountRepository,
TransactionRepository transactionRepository
)
```

Bu karar constructor injection ve dependency inversion prensiplerine uygun olması için alındı.

Böylece `BankService`:

```text
InMemory repository
PostgreSQL repository
Mock repository
```

gibi farklı implementationlarla çalışabilir.

---

### Karar 9: Spring Boot eklenmeden önce manuel dependency injection uygulanacak

Spring Boot henüz projeye eklenmeyecektir.

İlk aşamada repository nesneleri `Main` içinde manuel olarak oluşturulup `BankService` constructorına verilecektir.

Bu kararın amacı, Spring'in otomatik yaptığı dependency injection işlemini önce saf Java üzerinde anlamaktır.

---

### Karar 10: Main şimdilik inbound CLI adapter olarak kalacak

Mevcut `Main` classı silinmeyecektir.

Şimdilik:

```text
Main -> inbound CLI adapter
```

olarak çalışmaya devam edecektir.

Daha sonra Spring Boot eklendiğinde:

```text
REST Controller -> yeni inbound adapter
```

olacaktır.

Business logic BankService içinde kaldığı için CLI ve REST aynı application logic'i kullanabilecektir.

---

### Karar 11: Refactor kademeli yapılacak

Package, repository, Spring Boot ve PostgreSQL geçişleri tek adımda yapılmayacaktır.

Belirlenen sıra:

1. Package ayrımı
2. Repository interface'leri
3. In-memory repository implementationları
4. Constructor injection
5. Unit testlerin tekrar çalıştırılması
6. Spring Boot
7. REST Controller
8. Postman
9. PostgreSQL
10. Docker

Bu karar, hata kaynağını daha kolay tespit etmek ve projeyi mümkün olduğunca her aşamada çalışır durumda tutmak için alındı.

---

### Gün sonundaki mimari durum

- Domain modelleri ilgili package'lara taşındı.
- Application ve adapter package yapısı oluşturuldu.
- Repository outbound portları oluşturuldu.
- İlk in-memory repository adapterı yazıldı.
- BankService'in repository'lere geçirilmesi henüz tamamlanmadı.
- Spring Boot ve PostgreSQL entegrasyonuna henüz başlanmadı.

Bu aşama tamamlandığında `BankService`, verinin nerede ve nasıl saklandığını bilmeden çalışabilir hale gelecektir.
