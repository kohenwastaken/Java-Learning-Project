# Architecture Decision — İşlem Geçmişi Response Tasarımı

## Bağlam

Bir hesaba ait para yatırma, para çekme ve transfer işlemlerinin REST API üzerinden listelenmesi gerekiyordu.

Domain katmanındaki `Transaction` nesnesinin doğrudan JSON response olarak açılması, iç model ile API sözleşmesini birbirine bağlayabilirdi.

Ayrıca hiç işlemi olmayan bir hesap için dönecek response biçiminin belirlenmesi gerekiyordu.

## Karar

İşlem geçmişi şu endpoint üzerinden sunulacaktır:

```text
GET /api/accounts/{accountId}/transactions
```

Domain nesneleri doğrudan döndürülmeyecek, her işlem `TransactionResponse` DTO’suna çevrilecektir.

```text
List<Transaction>
        ↓
List<TransactionResponse>
```

Hiç işlem bulunmadığında response body olarak boş JSON listesi döndürülecektir:

```json
[]
```

`targetId`, para yatırma ve para çekme işlemlerinde bulunmayabileceği için nullable `Integer` olarak tutulacaktır.

## Gerekçe

- API sözleşmesini domain modelinden ayırmak
- Response alanlarını açıkça kontrol etmek
- Boş sonuçlarda `null` yerine güvenli ve standart bir liste döndürmek
- Bütün işlem türlerini aynı response modeli içinde göstermek

## Sonuçlar

### Olumlu

- İstemci her zaman JSON listesi alır.
- Domain modeli ileride değişse bile API response’u bağımsız tutulabilir.
- Para yatırma, para çekme ve transfer işlemleri tek endpoint altında görüntülenebilir.

### Maliyet

- Domain nesnelerinin DTO’ya dönüştürülmesi gerekir.
- Mevcut yapıda var olmayan hesap ile işlemi olmayan hesap aynı boş listeyi döndürebilir.
- Gelecekte `404 Not Found` ayrımı için hesap varlığı kontrolü veya merkezi hata yönetimi eklenmelidir.
