# Architecture Decision — Hesap İşlemleri için Endpoint Tasarımı

## Bağlam

Para yatırma, para çekme ve transfer işlemlerinin REST API üzerinden erişilebilir olması gerekiyordu. Bu işlemler hesap bakiyesini ve işlem geçmişini değiştirmektedir.

Endpoint adreslerinin, HTTP methodlarının ve request modellerinin tutarlı olması gerekmektedir.

## Karar

Hesap işlemleri `/api/accounts/{accountId}` kaynağı altında tanımlanacaktır.

```text
POST /api/accounts/{accountId}/deposits
POST /api/accounts/{accountId}/withdrawals
POST /api/accounts/{accountId}/transfers
```

Bakiye görüntüleme işlemi veri değiştirmediği için GET olarak kalacaktır:

```text
GET /api/accounts/{accountId}/balance
```

Her işlem için ayrı request DTO’su kullanılacaktır:

```text
DepositRequest
WithdrawRequest
TransferRequest
```

Transfer işleminde URL’deki `accountId` gönderen hesabı, request body içindeki `targetId` ise alıcı hesabı temsil edecektir.

Başarılı hesap işlemlerinde mevcut `BalanceResponse` kullanılarak etkilenen hesabın güncel bakiyesi döndürülecektir.

## Gerekçe

- Durum değiştiren işlemler için POST kullanılması
- Endpoint’lerin hesap kaynağı altında tutarlı biçimde gruplanması
- URL’den gelen gönderen hesabın body içinde tekrar edilmemesi
- DTO isimlerinin işlemin amacını açıkça göstermesi
- İşlem sonrasında güncel bakiyenin istemciye hemen bildirilmesi

## Sonuçlar

### Olumlu

- Endpoint yapısı okunabilir ve tahmin edilebilir hâle gelmiştir.
- Her request yalnızca ihtiyaç duyduğu alanları taşır.
- Controller ile service arasındaki veri akışı açık biçimde görülür.
- Postman testleri daha düzenli yürütülebilir.

### Maliyet

- Farklı hata türleri şimdilik çoğunlukla aynı `400 Bad Request` cevabına dönmektedir.
- Daha ayrıntılı HTTP hata modeli için merkezi exception handling gerekecektir.
