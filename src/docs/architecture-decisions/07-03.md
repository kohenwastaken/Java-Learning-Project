## architecture-decisions


### Büyük teknoloji geçişinden önce çekirdek kod düzeltilecek

#### Karar

Spring Boot, PostgreSQL, Docker ve hexagonal architecture aşamalarına geçmeden önce mevcut CLI uygulamasındaki buglar düzeltilecek.

#### Gerekçe

Mevcut mantık hatalıysa, bu hatalar daha büyük mimariye taşınır.

Önce para yatırma, para çekme, transfer ve log gibi temel işlemlerin doğru çalışması gerekir.

---

### Para validasyonları Account classında yapılacak

#### Karar

Negatif, sıfır veya bakiyeden fazla para işlemleri `Account` domaini içinde engellenecek.

#### Gerekçe

Bakiye ve para kuralları Account domain'ine aittir.

Ancak Account classı kullanıcı arayüzünden bağımsız kalmalıdır.

Account -> kural kontrolü
Main/accountMenu -> kullanıcı mesajları

---

### Account classı IO kullanmayacak

#### Karar

`Account` classı içinde `IO.println()` veya `Scanner` kullanılmayacak.

#### Gerekçe

Account classı konsol uygulamasına bağımlı olmamalıdır.

İleride uygulama Spring Boot, Postman veya web API üzerinden çalıştırıldığında Account classı yine kullanılabilir olmalıdır.

---

### İşlem sonucu boolean ile bildirilecek

#### Karar

Para yatırma, çekme ve gönderme methodları işlem sonucunu boolean olarak döndürecek.

#### Gerekçe

Bu yöntem `Account` classını sade tutar.

Mesaj gösterme veya log oluşturma kararı dış katmanda verilir.

---

### Transferde hedef hesap önce doğrulanacak

#### Karar

Para transferinde hedef hesap bulunmadan gönderen hesaptan para düşülmeyecek.

#### Gerekçe

Hedef hesap yoksa para düşülmesi ciddi veri tutarsızlığı oluşturur.

Bu yüzden transfer sırası şu şekilde olmalıdır:
```text
hedef ID al
kendine transfer mi kontrol et
hedef hesap var mı kontrol et
miktar geçerli mi kontrol et
gönderenden düş
alıcıya ekle
transaction oluştur
```

---

### Kendine para gönderme engellenecek

#### Karar

Kullanıcı kendi hesap ID'sine para gönderemeyecek.

#### Gerekçe

Bu işlem bakiyeyi değiştirmese bile gereksiz ve yanıltıcı transaction log oluşturur.

---

### ID sayaçları geçici olarak static tutulacak

#### Karar

`ID` ve `tID` class-level static değişken olarak tutulacak.

#### Gerekçe

Primitive `int` methoda parametre olarak gönderildiğinde kopyalanır.
Bu nedenle `accountMenu()` içinde artırılan `tID`, main tarafındaki değeri güncellemez.

Static sayaçlar CLI aşaması için pratik çözüm sağlar.

### Not

İleride bu sorumluluk veritabanı katmanına taşınmalıdır.

---

### Güvenli input için helper methodlar kullanılacak

#### Karar

Doğrudan `sc.nextInt()` ve `new BigDecimal(sc.nextLine())` kullanmak yerine helper methodlar kullanılacak.
```java
readInt(sc, message)
readBigDecimal(sc, message)
```
#### Gerekçe

Kullanıcı hatalı giriş yaptığında programın çökmesi engellenir.

Bu methodlar input'u String olarak alır, parse etmeyi dener ve hata olursa tekrar sorar.

---

### Kullanıcı inputları nextLine ile okunacak

#### Karar

Input akışında mümkün olduğunca `nextLine()` kullanılacak.

#### Gerekçe

`nextInt()` ve `nextLine()` birlikte kullanıldığında Enter karakteri yüzünden boş input problemi oluşabilir.

`nextLine()` + parse yaklaşımı daha kontrollü ve güvenlidir.

---

### BigDecimal inputunda virgül desteklenecek

#### Karar

Kullanıcının para girişinde virgül kullanması desteklenecek.
```java
input = input.replace(",", ".");
```
#### Gerekçe

Kullanıcılar alışkanlıkla decimal ayırıcı olarak virgül girebilir.

Bu dönüşüm kullanıcı deneyimini iyileştirir ve NumberFormatException riskini azaltır.
