# Architecture Decisions

## 1. HTTP girişleri için request ve response DTO’ları kullanılacak

### Karar

REST controller’ları, domain nesnelerini doğrudan request veya response olarak kullanmayacaktır.

Müşteri kayıt işlemi için ayrı DTO’lar kullanılacaktır:

```text
RegisterCustomerRequest
CustomerResponse
```

### Gerekçe

Kayıt isteğinde kullanıcıdan account ID alınmamalıdır. Bu değer sistem tarafından oluşturulmalıdır.

HTTP cevabında ise parola gibi hassas bilgiler gönderilmemelidir.

Request ve response modellerinin domain modelinden ayrılması, API sözleşmesinin daha kontrollü yönetilmesini sağlar.

### Sonuç

HTTP akışı şu şekilde olacaktır:

```text
JSON request
    ↓
Request DTO
    ↓
Domain/Application
    ↓
Response DTO
    ↓
JSON response
```

---

## 2. Basit DTO’lar Java record olarak tanımlanacak

### Karar

Yalnızca veri taşıyan ve oluşturulduktan sonra değiştirilmesi gerekmeyen request ve response DTO’ları Java `record` yapısıyla tanımlanacaktır.

### Gerekçe

Record kullanımı aşağıdaki tekrar eden kodları azaltır:

- Field tanımları
- Constructor
- Accessor methodları
- `equals()`
- `hashCode()`
- `toString()`

DTO’ların temel görevi veri taşımak olduğu için record yapısı bu amaçla uyumludur.

### Sonuç

Örnek kullanım:

```java
public record RegisterCustomerRequest(
        String name,
        String surname,
        String password
) {
}
```

Daha karmaşık davranış veya değiştirilebilir durum gerektiren nesneler normal class olarak tutulabilecektir.

---

## 3. REST controller yalnızca HTTP adaptasyonu yapacak

### Karar

CustomerController içerisinde müşteri oluşturma iş kuralları yazılmayacaktır.

Controller’ın sorumlulukları:

- HTTP isteğini almak
- Request DTO’yu okumak
- BankService’i çağırmak
- Sonucu response DTO’ya dönüştürmek
- HTTP cevabını üretmek

### Gerekçe

İş mantığının controller’a taşınması aynı kuralların CLI, REST veya başka giriş adapterlarında tekrarlanmasına neden olur.

Mevcut BankService zaten müşteri oluşturma use-case’ini yürütmektedir.

### Sonuç

Akış:

```text
CustomerController
    ↓
BankService.registerCustomer()
    ↓
Repository
```

şeklinde korunacaktır.

---

## 4. Müşteri oluşturma işlemi POST endpoint’i olarak sunulacak

### Karar

Yeni müşteri oluşturma işlemi aşağıdaki endpoint üzerinden sunulacaktır:

```text
POST /api/customers
```

### Gerekçe

POST, istemciden veri alarak yeni bir kaynak veya kayıt oluşturma işlemleri için uygundur.

Müşteri bilgileri request body içerisinde JSON olarak gönderilecektir.

### Sonuç

Tarayıcı adres çubuğundan gönderilen standart GET isteği bu işlemi çalıştırmayacaktır.

Endpoint, Postman veya başka bir HTTP istemcisi üzerinden POST yöntemi ve JSON body ile çağrılacaktır.

---

## 5. CLI ve REST aynı BankService katmanını kullanacak

### Karar

CLI Main ve REST controller’ları farklı use-case implementationları oluşturmayacaktır.

Her iki inbound adapter aynı BankService’i kullanacaktır.

### Gerekçe

Bankacılık kurallarının giriş yönteminden bağımsız olması istenmektedir.

Bu yapı aynı kuralların farklı yerlerde tekrar yazılmasını engeller.

### Sonuç

Main içerisindeki `while`, `switch`, `Scanner` ve çıktı kodları yalnızca CLI adapterına ait olacaktır.

REST tarafında aynı görevler Spring’in HTTP routing, request body ve response mekanizmalarıyla gerçekleştirilecektir.

---

## 6. API istekleri Postman collection içerisinde tutulacak

### Karar

Geliştirilen REST endpoint’leri Postman masaüstü uygulamasıyla test edilecek ve ilgili istekler bir collection içerisinde saklanacaktır.

### Gerekçe

Postman collection kullanımı:

- Endpoint’lerin tekrar kolayca çalıştırılmasını
- URL, HTTP yöntemi ve request body’nin saklanmasını
- API testlerinin düzenli tutulmasını
- Daha sonra paylaşılabilir bir test seti oluşturulmasını

sağlar.

### Sonuç

İlk saklanacak istek:

```text
Register Customer
POST http://localhost:8080/api/customers
```

olacaktır.
