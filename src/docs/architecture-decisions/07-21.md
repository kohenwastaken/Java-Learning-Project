# Architecture Decision — REST Inbound Adapter ve DTO Sınırı

## Bağlam

Bankacılık uygulaması başlangıçta CLI üzerinden kullanılmaktaydı. Aynı iş kurallarının HTTP üzerinden erişilebilir olması için Spring Boot REST katmanı eklendi.

Domain nesnelerinin ve servis methodlarının doğrudan dış dünyaya açılması, API sözleşmesi ile iç modelin birbirine fazla bağımlı hâle gelmesine neden olabilirdi.

## Karar

REST API, hexagonal mimaride bir inbound adapter olarak konumlandırılacaktır.

```text
adapter/in/rest
```

Controller sınıfları gelen HTTP isteklerini karşılayacak ve işlemleri `BankService` üzerinden yürütecektir.

İstek ve cevap verileri için domain nesneleri yerine ayrı DTO record’ları kullanılacaktır:

```text
RegisterCustomerRequest
CustomerResponse
BalanceResponse
```

Controller’lar iş kurallarını uygulamayacak; yalnızca HTTP verisini application katmanına aktaracak ve sonucu HTTP cevabına dönüştürecektir.

## Gerekçe

- Domain modelinin REST teknolojisinden bağımsız kalması
- Şifre gibi alanların response içinde yanlışlıkla açılmasının önlenmesi
- API sözleşmesinin kontrollü biçimde tanımlanması
- İstek ve cevap modellerinin okunabilir olması
- CLI ve REST adapterlarının aynı `BankService` katmanını kullanabilmesi

## Sonuçlar

### Olumlu

- Domain modeli dış API’den ayrılmıştır.
- REST response alanları açıkça kontrol edilebilir.
- Yeni bir frontend veya mobil istemci aynı API’yi kullanabilir.
- Controller katmanı sade kalır.

### Maliyet

- Domain modeli ile DTO arasında dönüşüm kodu gerekir.
- Her yeni API işlemi için yeni DTO oluşturmak gerekebilir.
