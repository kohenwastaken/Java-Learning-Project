## BankServiceTest, CustomerTest ve TransactionTest

Bugün unit test çalışmalarına devam edildi. Önceki gün `AccountTest` yazılmıştı. Bugün ise `BankService`, `Customer` ve `Transaction` classları için unit test mantığı üzerinde çalışıldı.

Ana hedef, sadece domain classlarını değil, service layer içindeki business flow'u da test edilebilir hale getirmekti.

---

### 1. BankServiceTest neden önemli?

`BankService`, projenin ana iş akışını yöneten class olduğu için unit test açısından en kritik classlardan biridir.

`AccountTest` daha çok tek bir hesabın bakiye kurallarını test ederken, `BankServiceTest` şu daha büyük akışları test eder:

```text
Müşteri kayıt ediliyor mu?
Login doğru çalışıyor mu?
Para yatırma sonucunda bakiye değişiyor mu?
Başarılı işlemde transaction oluşuyor mu?
Başarısız işlemde transaction oluşmuyor mu?
Transferde hem gönderen hem alıcı bakiye değişimleri doğru mu?
Transaction geçmişi doğru filtreleniyor mu?
```

Yani BankServiceTest, sadece tek bir methodun sonucunu değil, methodun yönettiği bütün iş akışını kontrol eder.

---

### 2. registerCustomer testleri

İlk olarak müşteri kayıt işlemi test edildi.

Test edilen davranışlar:

```text
registerCustomer çağrılınca Customer null olmamalı.
İlk customer ID değeri 1 olmalı.
Customer name doğru set edilmeli.
Customer surname doğru set edilmeli.
Yeni oluşturulan hesabın başlangıç bakiyesi 1000 olmalı.
```

Örnek:

```java
@Test
void registerCustomer_shouldCreateCustomerWithInitialBalance() {
BankService bankService = new BankService();

    Customer customer = bankService.registerCustomer("Arif", "Yılmaz", "123");

    assertNotNull(customer);
    assertEquals(1, customer.getAccId());
    assertEquals("Arif", customer.getName());
    assertEquals("Yılmaz", customer.getSurname());
    assertBigDecimalEquals(BigDecimal.valueOf(1000), bankService.showBalance(customer.getAccId()));
}
```

Bu testte 1000 değerine bağlı kalmak doğrudur. Çünkü bu testin amacı zaten yeni açılan hesabın başlangıç bakiyesini doğrulamaktır.

Ayrıca birden fazla müşteri kaydedildiğinde ID değerinin artıp artmadığı da test edildi.

```text
İlk müşteri ID: 1
İkinci müşteri ID: 2
```

Bu test, `userID++` mantığının doğru çalıştığını garanti eder.

### 3. loginAccount testleri

Login işlemi basit görünse de önemli bir güvenlik/business rule içerir.

Test edilen durumlar:

```text
Doğru ID + doğru şifre -> Customer dönmeli.
Doğru ID + yanlış şifre -> null dönmeli.
Sistemde olmayan ID -> null dönmeli.
```

Bu testler sayesinde login davranışının sadece mutlu senaryoda değil, hatalı senaryolarda da doğru çalıştığı kontrol edildi.

Örnek:

```java
@Test
void loginCustomer_ifPasswordIsWrong_shouldReturnNull() {
BankService bankService = new BankService();
Customer customer = bankService.registerCustomer("Arif", "Yılmaz", "123");

    Customer loginResult = bankService.loginAccount(customer.getAccId(), "wrong-password");

    assertNull(loginResult);
}
```

Bu test, ID doğru olsa bile şifre yanlışsa login işleminin başarısız olması gerektiğini doğrular.

### 4. Deposit işlemleri için BankService testleri

Para yatırma işlemi için hem başarılı hem başarısız senaryolar test edildi.

Başarılı senaryoda kontrol edilenler:

```text
DepositResult.SUCCESS dönmeli.
Bakiye yatırılan miktar kadar artmalı.
Transaction listesine 1 kayıt eklenmeli.
Transaction type DEPOSIT olmalı.
Transaction amount doğru olmalı.
Transaction sourceID customer ID olmalı.
Transaction targetID null olmalı.
```

Önemli bir test prensibi olarak, başlangıç bakiyesini doğrudan `1000` varsaymak yerine işlem öncesi bakiye değişkene alındı.

```java
BigDecimal balanceBefore = bankService.showBalance(customer.getAccId());
BigDecimal depositAmount = BigDecimal.valueOf(1000);
```

Sonra beklenen bakiye şöyle hesaplandı:

```java
balanceBefore.add(depositAmount)
```
Bu yaklaşım daha doğrudur. Çünkü testin amacı “başlangıç bakiyesi 1000 mi?” değil, “deposit amount kadar bakiye arttı mı?” sorusudur.

Başarısız senaryolarda ise şu kurallar test edildi:

```text
Negatif para yatırma INVALID_AMOUNT dönmeli.
0 TL yatırma INVALID_AMOUNT dönmeli.
Başarısız işlemde bakiye değişmemeli.
Başarısız işlemde transaction oluşmamalı.
```

---

### 5. Withdraw işlemleri için BankService testleri

Para çekme işlemi için de başarılı ve başarısız senaryolar test edildi.

Başarılı senaryoda:

```text
WithdrawResult.SUCCESS dönmeli.
Bakiye çekilen miktar kadar azalmalı.
Transaction oluşmalı.
Transaction type WITHDRAWAL olmalı.
Transaction targetID null olmalı.
```

Başarısız senaryolarda:

```text
Yetersiz bakiye -> INSUFFICIENT_BALANCE dönmeli.
Negatif miktar -> INVALID_AMOUNT dönmeli.
0 miktar -> INVALID_AMOUNT dönmeli.
Başarısız işlemde bakiye değişmemeli.
Başarısız işlemde transaction oluşmamalı.
```

Burada önemli nokta, sadece result kontrolü yapmakla yetinilmemesidir.

Örneğin yetersiz bakiye durumunda:

```text
Result doğru mu?
Bakiye aynı kaldı mı?
Transaction oluşmadı mı?
```

üçü birlikte kontrol edildi.

Bu, testin kalitesini artırır.

### 6. Transfer işlemleri için BankService testleri

Transfer, `BankService` içindeki en karmaşık işlem olduğu için daha kapsamlı test edildi.

Başarılı transferde kontrol edilenler:

```text
TransferResult.SUCCESS dönmeli.
Gönderenin bakiyesi amount kadar azalmalı.
Alıcının bakiyesi amount kadar artmalı.
Gönderenin transaction listesinde transfer görünmeli.
Alıcının transaction listesinde de aynı transfer görünmeli.
Transaction type TRANSFER olmalı.
Transaction amount doğru olmalı.
Transaction sourceID gönderen ID olmalı.
Transaction targetID alıcı ID olmalı.
```

Örnek mantık:

```java
BigDecimal senderBalanceBefore = bankService.showBalance(sender.getAccId());
BigDecimal receiverBalanceBefore = bankService.showBalance(receiver.getAccId());
BigDecimal sendAmount = BigDecimal.valueOf(500);
```

Beklenen bakiyeler:

```java
senderBalanceBefore.subtract(sendAmount)
receiverBalanceBefore.add(sendAmount)
```

Bu yaklaşım, testi başlangıç bakiyesine gereksiz bağımlı olmaktan kurtarır.

### 7. Transfer başarısız senaryoları

Transfer için şu başarısız durumlar test edildi:

```text
Alıcı hesap bulunamazsa -> ACCOUNT_NOT_FOUND
Kendi hesabına transfer yapılırsa -> INVALID_SELF_ID
Negatif amount girilirse -> INVALID_AMOUNT
0 amount girilirse -> INVALID_AMOUNT
Yetersiz bakiye varsa -> INSUFFICIENT_BALANCE
```

Bu durumlarda ortak olarak şunlar kontrol edildi:

```text
Result doğru dönmeli.
Gönderenin bakiyesi değişmemeli.
Alıcının bakiyesi değişmemeli.
Transaction oluşmamalı.
```

Bu testler, transfer işleminde başarısız senaryoların sisteme yan etki bırakmadığını garanti eder.

### 8. balanceBefore kullanımı

Bugün önemli bir test yazma prensibi öğrenildi:

```text 
Testin amacı başlangıç bakiyesini test etmek değilse, 1000 gibi sabit değerlere gereksiz bağımlı kalmamak gerekir.
```
Örneğin kötü yaklaşım:

```java
assertBigDecimalEquals(BigDecimal.valueOf(1000), bankService.showBalance(customer.getAccId()));
```
Bu bazı testlerde çalışır ama testin niyetini zayıflatır.

Daha iyi yaklaşım:

```java
BigDecimal balanceBefore = bankService.showBalance(customer.getAccId());

DepositResult result = bankService.depositToAccount(customer.getAccId(), BigDecimal.valueOf(-500));

assertBigDecimalEquals(balanceBefore, bankService.showBalance(customer.getAccId()));
```
Bu test artık şunu söyler:

```text
Başarısız işlemden sonra bakiye aynı kalmalı.
```

Bu daha doğru ve daha dayanıklı bir testtir.

### 9. Transaction kontrolü

Başarılı işlemlerde sadece bakiyeyi kontrol etmek yeterli değildir.

Çünkü `BankService` aynı zamanda transaction log oluşturmaktan da sorumludur.

Bu yüzden başarılı deposit, withdraw ve transfer işlemlerinde transaction detayları da kontrol edildi.

Örnek transaction kontrolleri:

```java
assertEquals(Transaction.TransactionType.TRANSFER, transaction.getType());
assertBigDecimalEquals(sendAmount, transaction.getAmount());
assertEquals(sender.getAccId(), transaction.getSourceID());
assertEquals(receiver.getAccId(), transaction.getTargetID());
```

Bu testler sayesinde transaction log sisteminin doğru çalıştığı da garanti edilir.

### 10. CustomerTest

`Customer` classı daha basit bir class olduğu için kısa testler yeterlidir.

Test edilen davranışlar:

```text
Constructor verilen değerlerle Customer oluşturmalı.
getAccId doğru ID dönmeli.
getName doğru name dönmeli.
getSurname doğru surname dönmeli.
passwordMatches doğru şifrede true dönmeli.
passwordMatches yanlış şifrede false dönmeli.
```

Örnek:

```java
@Test
void passwordMatches_whenPasswordIsCorrect_shouldReturnTrue() {
Customer customer = new Customer("Arif", "Yılmaz", "123", 1);

    assertTrue(customer.passwordMatches("123"));
}
```

Bu testlerde dikkat edilen nokta:

```text
Password için getter yazılmadı.
Şifre dışarıdan okunmadı.
Sadece passwordMatches davranışı test edildi.
```

Bu, encapsulation açısından daha doğru bir yaklaşımdır.

### 11. TransactionTest

`Transaction` classı daha çok immutable data taşıyan bir class gibi davrandığı için testleri basittir.

Test edilen davranışlar:

```text
Constructor verilen transactionID değerini set etmeli.
Transaction type doğru set edilmeli.
Amount doğru set edilmeli.
SourceID doğru set edilmeli.
TargetID doğru set edilmeli.
TargetID null olabilir.
```

Örnek:

```java
@Test
void constructor_shouldCreateTransactionWithGivenValues() {
Transaction transaction = new Transaction(
        1,
        Transaction.TransactionType.TRANSFER,
        BigDecimal.valueOf(500),
        1,
        2
    );

    assertEquals(1, transaction.getTransactionID());
    assertEquals(Transaction.TransactionType.TRANSFER, transaction.getType());
    assertBigDecimalEquals(BigDecimal.valueOf(500), transaction.getAmount());
    assertEquals(1, transaction.getSourceID());
    assertEquals(2, transaction.getTargetID());
}
```

Deposit ve withdrawal işlemlerinde `targetID` null olabilir. Bu yüzden null targetID durumu ayrıca test edilebilir.

### 12. Testlerde tekrar eden yardımcı method

`BigDecimal` karşılaştırması için test classlarında helper method kullanılmaya devam edildi.

```java
private void assertBigDecimalEquals(BigDecimal expected, BigDecimal actual) {
    assertEquals(0, expected.compareTo(actual));
}
```

Bu helper sayesinde:

```text
BigDecimal scale farklarından kaynaklı hatalar engellenir.
Testler daha okunabilir olur.
Aynı karşılaştırma mantığı tekrar tekrar yazılmaz.
```

### 13. Bugünkü kazanımlar

Bugün öğrenilen ve uygulanan ana konular:

- BankService gibi service layer classları nasıl test edilir?
- Sadece result değil, state değişimi de nasıl test edilir?
- Başarılı işlemlerde transaction oluşumu nasıl doğrulanır?
- Başarısız işlemlerde transaction oluşmadığı nasıl doğrulanır?
- Başarısız işlemlerde bakiyenin aynı kaldığı nasıl test edilir?
- balanceBefore kullanımı neden daha sağlıklı bir test yaklaşımıdır?
- Customer gibi basit domain classları nasıl test edilir?
- Password getter yerine passwordMatches davranışı nasıl test edilir?
- Transaction gibi immutable data classları nasıl test edilir?
