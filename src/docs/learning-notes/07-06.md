## Service layer refactor

### Hexagonal architecture mantığı

Hexagonal architecture bir kütüphane değil, kodu düzenleme yaklaşımıdır.

Temel fikir:

```text
İş mantığı dış dünyaya bağımlı olmamalı.
CLI, REST API veya PostgreSQL değişse bile core logic mümkün olduğunca değişmemeli.
```

```text
CLI / REST Controller  -> inbound adapter
Service                -> application layer
Domain classes         -> Customer, Account, Transaction
Repository             -> veri erişim katmanı
PostgreSQL / ArrayList -> dış veri kaynağı
```

---

### Service layer nedir?

Service layer uygulamanın iş akışını yöneten katmandır.

Örneğin:

```text
Kullanıcı kayıt olur
Kullanıcı giriş yapar
Para yatırılır
Para çekilir
Para transfer edilir
Transaction log tutulur
```
Bu işlemler `Main` içinde değil, service class içinde yönetilmeye başlanmalıdır.

---

### Main ve Service ayrımı

Bugün şu ayrım netleşti:

#### Main / CLI:
- Menü gösterir
- Kullanıcıdan input alır
- Service methodlarını çağırır
- Sonucu ekrana basar

#### BankService:
- Customer oluşturur
- Account oluşturur
- Login kontrolü yapar
- Para işlemlerini yönetir
- Transaction kayıtlarını tutar

Service içinde `Scanner` veya `IO.println()` olmamalıdır

---

### BankService classı

İlk aşamada çok sayıda service class açmak yerine tek bir `BankService` classı ile başlanmasına karar verildi.

`BankService` içinde geçici olarak şu state'ler tutulabilir:
```java
private List<Customer> customerList = new ArrayList<>();
private List<Account> accountList = new ArrayList<>();
private List<Transaction> transactionList = new ArrayList<>();

private int userID = 1;
private int transactionID = 1;
```

Bu yapı `Main`'in yükünü azaltır.

---

### List ve ArrayList farkı

`List` bir interface, `ArrayList` ise bu interface'in implementasyonudur.

Daha esnek kullanım:
```java
private List<Customer> customerList = new ArrayList<>();
```

Bu şu anlama gelir:

Kod sadece liste davranışına ihtiyaç duyuyor.
Arkada şu an ArrayList kullanılıyor.
İleride başka bir List implementasyonu kullanılabilir.

Bu yüzden field veya parametre tiplerinde `List`, nesne oluştururken `ArrayList` kullanmak daha esnektir.

---

### Static field yerine instance field

`BankService` içinde listeleri ve ID sayaçlarını static yapmak yerine instance field olarak tutmak daha doğru görüldü.

Yani:
```java
BankService bankService = new BankService();
```

oluşturulur ve işlemler bu nesne üzerinden yürütülür.

Bu yaklaşım global değişkenlerden daha kontrollüdür.

---

### Registration refactor

Eski yapıda `registry()` hem input alıyor hem `Customer` oluşturuyor hem de Account oluşturma akışına karışıyordu.

Yeni düşünce:

```text
registry():
- ad soyad alır
- şifre alır
- BankService'e gönderir

BankService.registerCustomer():
- ID üretir
- Customer oluşturur
- Account oluşturur
- listelere ekler
- oluşturulan Customer'ı döndürür
```

Bu sayede `Customer` ve `Account` oluşturma işi Main'den BankService'e taşınır.

---

### registerCustomer dönüş değeri

`registerCustomer()` methodunun `void` olmaması daha faydalı görüldü.

Çünkü kayıt sonrası Main tarafının kullanıcıya ID göstermesi gerekir.

Bu yüzden method başarılı kayıt sonrası `Customer` döndürebilir.

Customer dönerse:
```text
- kayıt başarılıdır
- accID okunabilir
- kullanıcıya ID gösterilebilir
```

---

### Login refactor

Login işlemi de ikiye ayrılmaya başlandı.

#### CLI tarafı:

- Kullanıcıdan ID alır
- Kullanıcıdan şifre alır
- BankService.loginAccount(...) çağırır

#### BankService tarafı:

- Customer listesinde ID ve şifre eşleşmesi arar
- başarılıysa Customer döndürür
- başarısızsa null döndürür

Bu ayrım service mantığına uygundur çünkü `BankService` kullanıcıyla konuşmaz, sadece doğrulama yapar.

---

## Bugün öğrenilen ana noktalar
- Hexagonal architecture kodu dış dünyadan bağımsız tutma yaklaşımıdır
- Service layer iş akışını yönetir
- Main sadece CLI/UI sorumluluğu taşımalıdır
- Service içinde Scanner ve IO.println() olmamalıdır
- Listeler Main yerine BankService içinde tutulabilir
- List kullanmak ArrayListe doğrudan bağımlılığı azaltır
- BankService için instance field kullanmak static/global değişkenden daha temizdir
- Registration ve login işlemleri input alma + business logic olarak ayrılmalıdır
