## Package Yapısı, Repository Pattern, Map ve Dependency Injection

Bugün projeyi CLI tabanlı ilk sürümden hexagonal architecture yapısına hazırlamak için çalışmalara başlandı.

Öncelikle Controller, Service, Repository ve Dependency Injection kavramları araştırıldı. Ardından mevcut projede bu kavramların nasıl uygulanacağı üzerinde çalışıldı.

---

### 1. Controller, Service ve Repository sorumlulukları

Katmanların temel sorumlulukları şu şekilde öğrenildi:

#### Controller

Controller, uygulamaya dışarıdan gelen istekleri karşılayan katmandır.

Görevleri:

- Kullanıcıdan veya HTTP isteğinden veri almak
- Gelen veriyi uygun service methoduna göndermek
- Service sonucunu kullanıcıya veya istemciye döndürmek

Mevcut CLI sürümünde `Main`, tam anlamıyla bir REST Controller olmasa da sisteme giriş noktası olduğu için inbound adapter benzeri bir görev üstlenmektedir.

Spring Boot aşamasında CLI yerine REST controllerlar oluşturulacaktır.

#### Service

Service, uygulamanın kullanım senaryolarını ve iş akışlarını yönetir.

Projede `BankService` şu işlemleri yönetmektedir:

- Müşteri kaydı
- Login
- Para yatırma
- Para çekme
- Transfer
- Transaction oluşturma
- Hesap hareketlerini listeleme

Service, kullanıcıdan doğrudan input almamalı ve verinin nasıl saklandığını bilmemelidir.

#### Repository

Repository, veriye erişim işlemlerini soyutlar.

Service şu tür komutlar verir:

```text
Müşteriyi kaydet.
Hesabı ID ile bul.
Transaction'ı kaydet.
Hesabın transactionlarını getir.
```

Bu işlemlerin ArrayList, `Map` veya `PostgreSQL` ile yapılması repository implementationının sorumluluğudur.

### 2. Package yapısı oluşturuldu

Projede bütün classlar daha önce aynı package altında bulunuyordu:

```text
org.example
```

Bugün classlar sorumluluklarına göre farklı package'lara taşındı.

```text
org.example.bank.domain.model

org.example.bank.domain.result

org.example.bank.application.service

org.example.bank.appliaction.port.out

org.example.bank.adapter.in.cli

org.example.bank.adapter.out.memory
```

Package'ların anlamı:

```text
domain
-> Bankacılık modelleri ve kuralları

application
-> Uygulamanın use-case ve iş akışları

adapter.in
-> Uygulamaya dışarıdan giriş yapan yapılar

adapter.out
-> Veritabanı veya bellek gibi dış sistemlere erişen yapılar
```

### 3. Package-private erişim problemi

Classlar aynı package içindeyken erişim belirleyicisi yazılmayan methodlar birbirlerine erişebiliyordu.

Örnek:

```java
boolean moneyDeposit(BigDecimal amount)
```

Bu method package-private'tır.

`Account` ve `BankService` farklı package'lara taşındığında `BankService`, bu methoda erişemez hale geldi.

Bu nedenle farklı katmanlardan çağrılması gereken methodlar `public` yapıldı.

Örnek:

```java
public boolean moneyDeposit(BigDecimal amount)
public WithdrawResult moneyWithdraw(BigDecimal amount)
public TransferResult moneySend(BigDecimal amount)
public boolean moneyReceive(BigDecimal amount)
```

Aynı şekilde `Main` tarafından çağrılan `BankService` use-case methodları da `public` olmalıdır.

Sadece class içinde kullanılan helper methodlar `private` kalmalıdır.

Örnek:

```java
private Account findAccountById(int accountId)
```

### 4. Repository interface'leri

Bugün repository interface'leri oluşturulmaya başlandı.

Oluşturulan interface'ler:

```text
AccountRepository
CustomerRepository
TransactionRepository
```

Repository interface yalnızca hangi işlemlerin yapılabileceğini tanımlar.

Örnek:

```java
public interface AccountRepository {

    Account save(Account account);

    Optional<Account> findById(int accountId);
}
```

Interface şu konularla ilgilenmez:

```text
Veri ArrayList'te mi tutuluyor?
Map kullanılıyor mu?
PostgreSQL bağlantısı var mı?
Hangi SQL sorgusu çalıştırılıyor?
```

Bunlar repository implementationının sorumluluğudur.

---

### 5. Repository interface ve implementation farkı

Interface, yapılabilecek işlemlerin sözleşmesidir.

```text
AccountRepository
- Account kaydedilebilir.
- Account ID ile bulunabilir.
```

Implementation ise bu işlemlerin nasıl gerçekleştirileceğini belirler.

```text
InMemoryAccountRepository
- Accountları bellekte Map içinde saklar.

İleride PostgreSqlAccountRepository
- Accountları PostgreSQL içinde saklar.

```
Böylece `BankService`, verinin nerede tutulduğunu bilmeden aynı repository interface'ini kullanabilir.

---

### 6. InMemoryAccountRepository

Bugün ilk repository implementationlarından biri olarak `InMemoryAccountRepository` oluşturuldu.

Örnek yapı:

```java
public class InMemoryAccountRepository implements AccountRepository {

    private final Map<Integer, Account> accounts = new HashMap<>();

    @Override
    public Account save(Account account) {
        accounts.put(account.getAccId(), account);
        return account;
    }

    @Override
    public Optional<Account> findById(int accountId) {
        return Optional.ofNullable(accounts.get(accountId));
    }
}
```

Bu class:

- `AccountRepository` interface'ini uygular
- Accountları geçici olarak bellekte saklar
- Account ID değerini anahtar olarak kullanır
- ID ile doğrudan Account bulabilir

### 7. Map nedir?

Map, verileri anahtar-değer ilişkisiyle saklayan bir koleksiyondur.

Örnek:

```text
1 -> Account nesnesi
2 -> Account nesnesi
3 -> Account nesnesi
```

Projede:

```text
Key   -> Account ID
Value -> Account
```
kullanılmaktadır.

Tanımlama:

```java
Map<Integer, Account>
```

Burada:

```text
Integer -> anahtar tipi
Account -> değer tipi
```

`Map` bir interface, `HashMap` ise bu interface'in implementationlarından biridir.

Bu kullanım:

```java
Map<Integer, Account> accounts = new HashMap<>();
```

şu yapıya benzer:

```java
List<Account> accounts = new ArrayList<>();
```

İlkinde `Map` interface, `HashMap` implementationdır.

İkincisinde `List` interface, `ArrayList` implementationdır.

---

### 8. Map üzerinde kullanılan temel methodlar

Bugün özellikle iki Map methodu öğrenildi.

#### put
```java
accounts.put(account.getAccId(), account);
```

Map içine anahtar ve değer ekler.

Örneğin Account ID 5 ise:

```text
5 -> Account nesnesi
```

şeklinde saklanır.

#### get
```java
accounts.get(accountId);
```

Verilen anahtara ait değeri getirir.

Eğer anahtar Map içinde yoksa `null` döner.

İleride öğrenilecek diğer temel methodlar:
```java
containsKey()
remove()
size()
values()
```

---

### 9. ArrayList ve Map farkı

`ArrayList`, elemanları sırayla saklar.

Bir Account'ı ID ile bulmak için bütün liste dolaşılabilir:

```java
for (Account account : accountList) {
    if (account.getAccId() == accountId) {
        return account;
    }
}
```

Map kullanıldığında doğrudan anahtarla erişilir:

```java
accounts.get(accountId);
```

Genel kullanım farkı:

```text
List
-> Elemanları sırayla tutmak ve dolaşmak için uygundur.

Map
-> Bir anahtar kullanarak belirli elemana erişmek için uygundur.
```

Account ve Customer nesneleri ID ile arandığı için Map kullanımı uygundur.

### 10. Optional kavramı

Repository `findById()` methodu doğrudan `Account` yerine şu tipi döndürmektedir:

```java
Optional<Account>
```

Bu şu anlama gelir:

```text
Account bulunabilir.
Account bulunamayabilir.
```

Kullanılan yapı:

```java
Optional.ofNullable(accounts.get(accountId))
```

Eğer Map içinden Account gelirse dolu Optional döner.

Eğer Map içinden `null` gelirse:

```java
Optional.empty()
```

döner.

Optional, methodun sonuç döndürmeyebileceğini açıkça ifade eder ve kontrolsüz `null` kullanımını azaltır.

### 11. Dependency Injection

Dependency Injection, bir classın ihtiyaç duyduğu nesneleri kendi içinde oluşturmak yerine dışarıdan almasıdır.

Eski `BankService` kendi listelerini oluşturuyordu:

```java
private final List<Account> accountList = new ArrayList<>();
```

Yeni yapıda `BankService` repository interface'lerine bağımlı olacaktır:

```java
private final AccountRepository accountRepository;
private final CustomerRepository customerRepository;
private final TransactionRepository transactionRepository;
```

Bu repository'ler constructor üzerinden verilecektir:

```java
public BankService(
CustomerRepository customerRepository,
AccountRepository accountRepository,
TransactionRepository transactionRepository
) {
this.customerRepository = customerRepository;
this.accountRepository = accountRepository;
this.transactionRepository = transactionRepository;
}
```

Buna constructor injection denir.

Şimdilik repository nesnelerini `Main` oluşturacaktır. Spring Boot aşamasında bu işlemi Spring'in IoC container yapısı gerçekleştirecektir.

### 12. Controller şu anda neden oluşturulmadı?

Bugün Controller mantığı araştırıldı ancak projeye henüz Spring REST Controller eklenmedi.

Şu anda:

```text
Main
-> CLI üzerinden sisteme giriş noktasıdır.
```

Spring Boot aşamasında:

```text
REST Controller
-> HTTP isteklerinin sisteme giriş noktası olacaktır.
```

Repository refactoru tamamlanmadan Spring Controller eklememeye karar verildi.

Önce saf Java üzerinde:

- Repository interface
- In-memory adapter
- Dependency injection
- BankService refactoru

tamamlanacaktır.
