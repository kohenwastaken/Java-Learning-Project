# Learning notes

## Genel bug düzeltme aşaması

Bugün Spring Boot / PostgreSQL / Docker gibi sonraki aşamalara geçmeden önce mevcut CLI uygulamasındaki temel mantık hatalarını düzeltmeye odaklanıldı.

Öncelik, banka işlemlerinin güvenli ve mantıksal olarak doğru çalışmasıydı.

### Account class içinde validation

Para yatırma, para çekme ve para gönderme işlemlerinde negatif veya sıfır miktar girilmesi engellenmeye başlandı.

Bu kontrollerin `Account` domaini içinde yapılması daha doğru görüldü çünkü bakiye kuralları Account classına aittir.

Ancak `Account` classı içinde `IO.println()` kullanılmadı.  
Kullanıcıya mesaj gösterme işi `Main/accountMenu` tarafında bırakıldı.

Öğrenilen ayrım:

```text
Account -> işlem kurallarını uygular
Main/accountMenu -> kullanıcıdan input alır ve mesaj gösterir
```

### boolean return mantığı

`moneyDeposit`, `moneyWithdraw` ve `moneySend` gibi methodlar `true` veya `false` döndürecek şekilde düzenlendi.

Bu sayede Account classı konsola mesaj yazmadan işlem sonucunu dışarıya bildirebilir.

### Transfer güvenliği

Para gönderme işleminde bazı kontroller eklendi:

```text
- Kullanıcı kendine para gönderememeli
- Hedef hesap sistemde yoksa işlem yapılmamalı

- Transfer başarılıysa transaction log oluşturulmalı
```

Bu kontroller sayesinde para gönderme işlemi daha güvenli hale getirildi.

### tID ve ID sayaçları

`transactionID` ve `account/customer ID` için kullanılan sayaçların methodlara primitive `int` olarak gönderilmesinin sorun oluşturabileceği fark edildi.

Çünkü Java'da primitive değerler methodlara kopyalanarak gider.

Bu yüzden `tID` ve `ID` class-level static değişken olarak tutuldu.

Bu karar mevcut CLI aşaması için pratik bir çözümdür.
İleride bu sorumluluk servis veya veritabanı katmanına taşınabilir.

---

### Güvenli input okuma

Kullanıcı string girdiğinde programın çökmesini engellemek için `sc.nextInt()` yerine daha güvenli helper methodlar yazıldı.

Menü ve ID girişleri için:

```java
readInt(sc, message)
```

Para miktarları için:

```java
readBigDecimal(sc, message)
```

kullanılmaya başlandı.

Bu methodlar input'u önce `String` olarak alır, sonra parse etmeyi dener.

---

### try-catch kullanımı

Hatalı sayı girişlerinde programın çökmesini engellemek için `try-catch` kullanıldı.

Örneğin kullanıcı sayı yerine harf girerse `NumberFormatException` oluşur.
Bu hata yakalanarak kullanıcıya tekrar giriş yaptırılır.

Öğrenilen mantık:
```text
input al
parse etmeyi dene
başarılıysa değeri döndür
hatalıysa kullanıcıya tekrar sor
```

---

### Virgüllü para girişi

Java `BigDecimal` String constructor'ı decimal ayırıcı olarak nokta bekler.
```text
99.5  -> geçerli
99,5  -> normalde hata verir
```
Bu yüzden input içinde virgül noktaya çevrildi:

```java
input = input.replace(",", ".");
```
Böylece kullanıcı `99,5` yazsa bile program bunu `99.5` olarak işleyebilir.

---

### nextInt / nextLine probleminin azaltılması

Daha önce `nextInt()` sonrası `nextLine()` kullanımı Enter karakteri yüzünden sorun çıkarabiliyordu.

Yeni yaklaşımla kullanıcı girişleri çoğunlukla `nextLine()` üzerinden alınmaya başladı.

Bu, input akışını daha tutarlı hale getirdi.

---

## Bugün öğrenilen ana noktalar
- Büyük teknoloji aşamasına geçmeden önce çekirdek mantığın sağlamlaştırılması gerekir
- Validation class içinde olabilir, ama kullanıcı mesajları UI/CLI tarafında kalmalı
- Account classı bakiye kurallarını korumalı
- boolean return ile işlem sonucu dışarı bildirilebilir
- Primitive `int` methoda kopyalanarak gider
- static sayaçlar CLI aşamasında pratik çözüm olabilir
- `try-catch` program çökmesini engellemek için kullanılabilir
- Güvenli input için `nextLine()` + parse yaklaşımı daha kontrollüdür
- `BigDecimal` inputlarında virgül/nokta farkı önemlidir
- Transfer işleminde hedef hesap kontrolü para düşmeden önce yapılmalıdır
