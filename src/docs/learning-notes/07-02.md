## Learning Notes

### accID Tipi: int mi String mi?

Bugün `accID` değerinin `int` mi yoksa `String` mi olması gerektiğini düşündüm.

Şu an `accID` sistem tarafından otomatik artırılan bir kimlik değeri olarak kullanılıyor:

```java
int ID = 0;
int accID = ID++;
```

Bu yüzden `int` kullanmak daha mantıklı.

### BigDecimal kullanımı

Para işlemleri için `double` veya `float` yerine `BigDecimal` kullanmanın daha doğru oldğuunu öğrendim.
Çünkü para işlemlerinde hassasiyet önemlidir. `double` beklenmeyen küsurat problemlerine yol açabilir.

```java
BigDecimal balance;
BigDecimal amount = new BigDecimal(sc.nextLine());
```
toplama ve çıkarma işlemler `+` veya `-` ile değil, methodlarla yapılır.

```java
balance = balance.add(amount);
balance = balance.subtract(amount);
```

karşılaştırma için ise `<` , `>` yerine `compareTo()` kullanılır.

```java
balance.compareTo(amount) >= 0
```

### Account class oluşturma

Bakiye ve hsap bilgilerini `Customer` içinde tutmak yerine ayrı class oluşturuldu.

```java
public class Account {
    int accID;
    BigDecimal balance;

    public Account(int accID, BigDecimal balance) {
        this.accID = accID;
        this.balance = balance;
    }
}
```

Bu sayede `Customer` kullanıcı bilgilerini, `Account` ise para ve bakiye bilgilerini temsil eder.

### Customer ve Account ilişkisi

Customer ile Account şimdilik ortak `accID` üzerinden ilişkilendiriliyor.

`Customer.accID = Account.accID`

Login işlemi Customer üzerinden yapılır. Giriş başarılı olduktan sonra ilgili Account, `accountList` içinde aranır.

### Referans Mantığı

`findAffiliatedAccount()` ile bulunan Account nesnesi yeni bir kopya değildir, `accountList` içindeki nesnenin referansıdır.

### Account içine davranış ekleme

Para yatırma işlemi Account classı içine method olarak taşınmaya başlandı.

```java
void addDeposit(BigDecimal amount) {
    this.balance = this.balance.add(amount);
}
```

Konsola yazma ve kullanıcıdan veri alma işlemleri Account içinde tutulmalı, Account sadece bakiye mantığını yönetmeli.

## Transaction / Log Sistemi

### Transaction class fikri

Hesap hareketlerini tutmak için ayrı bir `Transaction` classı oluşturma fikri geliştirildi.

Amaç: Her para hareketini ayrı bir nesne olarak saklamak.

Bir transaction içinde şu bilgiler tutulabilir:

- işlem ID
- işlem tipi
- miktar
- gönderen hesap ID
- hedef hesap ID

Bu sayede log bilgileri primitive değişkenlerle dağınık tutulmak yerine tek bir nesnede toplanır.

---

### Tek class mı, interface mi?

İlk olarak farklı işlem tipleri için interface ve farklı classlar düşünüldü.

Ancak mevcut aşamada tek bir `Transaction` classı yeterli görüldü.

Çünkü para yatırma, para çekme ve para gönderme işlemleri farklı görünse de log açısından hepsi ortak bir yapıya sahiptir:

```text
Bir işlem oldu.
Bir miktar para değişti.
Bir işlem tipi var.
İlgili hesap ID'leri var.
```
Bu yüzden şu an `Transaction class + TransactionType enum` yapısı tercih edildi.

---

### Enum kullanımı

işlem tiplerini String ile yazmak yerine enum kullanmak daha güvenli görüldü.

Örnek işlem tipleri:

```java
public enum TransactionType {
    DEPOSIT,
    WITHDRAWAL,
    SENT,
    RECEIVED
}
```

Enum kullanırken constructora String değil, enum değeri gönderilir.

`Transaction.TransactionType.DEPOSIT`

---

### Transaction constructor mantığı

Transaction bilgisi oluşturulurken, işlem bilgileri constructor ile verilir.

```java
public Transaction(
        int transactionID,
        TransactionType type,
        BigDecimal amount,
        int sourceID,
        Integer targetID
) {
    this.transactionID = transactionID;
    this.type = type;
    this.amount = amount;
    this.sourceID = sourceID;
    this.targetID = targetID;
}
```

Bu sayede işlem oluşturulduğu anda tüm temel bilgiler belli olur.

---

### TransactionID yönetimi

`transactionID` şu an bir sayı olarak tutulabilir.

```java
int tID = 0;
```

yani işlem oluşturulurken:

```java
Transaction t = new Transaction(tID++, ...);
```
mantığı kullanılabilir. Bu yöntem mevcut aşama için yeterlidir.

---

### targetID neden Integer?

Bazı işlemlerde hedef hesap vardır, bazılarında yoktur.

Örneğin:
```text
Para yatırma  -> hedef hesap yok
Para çekme    -> hedef hesap yok
Para gönderme -> hedef hesap var
```

Bu yüzden `targetID` için `int` yerine `Integer` değer kullanıldı.

Çünkü Integer `null` olabilir. hedef olmayan hesap işlemlerinde
```java
targetID = null;
```
denebilir.

---

### Para yatırma logu oluşturma

Para yatırma işleminden sonra bir Transaction nesnesi oluşturulup listeye eklenir.

```java
Transaction t = new Transaction(
        tID++,
        Transaction.TransactionType.DEPOSIT,
        depositAmount,
        loggedInAccount.accID,
        null
);

transactionList.add(t);
```

Burada:

```text
sourceID = işlem yapılan hesap
targetID = null
```

çünkü para yatırmada karşı hesap yoktur.

---

### Transferde tek log mu iki log mu?

Para gönderme işleminde iki ayrı log yerine tek Transaction kaydı yeterli görüldü.

Çünkü tek kayıt şu bilgiyi tamamen anlatabilir:

```text
sourceID hesabından targetID hesabına amount kadar para gönderildi.
```

Hesap hareketleri gösterilirken bakış açısına göre yorum yapılabilir. Ama şimdiki haliyle yeterlidir.

---

### Logları listeleme

Hesap hareketlerini göstermek için `transactionList` gezilir.

Giriş yapan hesap işlemde gönderen veya alıcı ise ekrana basılır.

```java
if (tr1.sourceID == loggedInAccount.accID ||
        (tr1.targetID != null && tr1.targetID == loggedInAccount.accID)) {
    // log yazdır
}
```

---

### null kontrolü

`targetID` bazı işlemlerde null olabilir.

Bu yüzden şu kullanım risklidir:

```java
tr1.targetID == loggedInAccount.accID
```

Çünkü targetID `null` ise hata oluşabilir.

Doğru yaklaşım:

```java
tr1.targetID != null && tr1.targetID == loggedInAccount.accID
```
önce null kontrolü yaptırılır sonra karşılaştırılır.

---
