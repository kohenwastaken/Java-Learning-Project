## 1. `@Override` Annotation’ı

`@Override`, yalnızca bir üst classın methodunu değiştirdiğimiz durumlarda kullanılmaz.

İki temel kullanım alanı vardır:

### Class kalıtımı

Bir class başka bir classı `extends` ile miras aldığında, üst classtan gelen bir methodu yeniden tanımlayabilir.

```java
class Animal {

    void makeSound() {
        System.out.println("Ses");
    }
}

class Cat extends Animal {

    @Override
    void makeSound() {
        System.out.println("Miyav");
    }
}
```

### Interface implementationı

Bir class bir interface’i implements ettiğinde, interface içinde tanımlanan methodları gerçekleştirmek zorundadır.

```java
public interface AccountRepository {

    Account save(Account account);
}
public class InMemoryAccountRepository implements AccountRepository {

    @Override
    public Account save(Account account) {
        return account;
    }
}
```

Buradaki `@Override`, yazılan methodun interface sözleşmesini uyguladığını belirtir.

`@Override` kullanmanın önemli faydası, method imzasındaki yazım hatalarının derleyici tarafından yakalanmasıdır.

Örneğin interface içinde:

```java
findByAccountId(int accountId)
```

varken implementation içinde yanlışlıkla:

```java
findAccountById(int accountId)
```

yazılırsa, `@Override` annotation’ı bu methodların eşleşmediğini gösterir.

## 2. Interface ve Implementation Arasındaki Sözleşme

Repository interface’lerinde yalnızca ihtiyaç duyulan işlemler tanımlandı.

Örneğin:

```java
public interface CustomerRepository {

    Customer save(Customer customer);

    Optional<Customer> findById(int customerId);
}
```

Interface şu sorulara cevap verir:

```text
Bir Customer nasıl kaydedilebilir?
Bir Customer ID ile nasıl aranabilir?
```

Ancak verinin nerede ve nasıl tutulacağını belirtmez.

Gerçek saklama yöntemi implementation tarafından belirlenir:

```java
public class InMemoryCustomerRepository
implements CustomerRepository {
}
```

Interface ve implementation methodlarının şu özellikleri uyumlu olmalıdır:

```text
Method adı
Parametre sayısı
Parametre türleri
Dönüş tipi
```

Aksi hâlde implementation, interface sözleşmesini yerine getirmiş sayılmaz.

## 3. Neden Repository İsmi Kullanıldı?

Repository, verinin saklandığı teknolojiyi değil, domain nesnelerine erişim sözleşmesini temsil eder.

Örneğin:

```java
AccountRepository
```

şunu ifade eder:

```text
Account kaydedilebilir.
Account aranabilir.
Account verisine erişilebilir.
```

Ancak şu detayları belirtmez:

```text
HashMap kullanılacak mı?
PostgreSQL kullanılacak mı?
Dosyaya mı yazılacak?
Bir API üzerinden mi alınacak?
```

Bu nedenle `AccountDatabase` gibi bir isim tercih edilmedi.

Database ismi, implementationın mutlaka gerçek bir veritabanı kullandığını düşündürürdü. Ancak mevcut implementation yalnızca bellekte veri tutmaktadır:

`InMemoryAccountRepository`

İleride aynı interface için farklı implementationlar yazılabilir:

- InMemoryAccountRepository
- PostgreSqlAccountRepository
- JpaAccountRepository
- FileAccountRepository

`BankService` yalnızca `AccountRepository` interface’ini bildiği için saklama teknolojisi değişse bile iş mantığının değişmesi gerekmez.

## 4. Package Yapısında `in` ve `out`

Oluşturulan temel package yapısı:

```text
org.example.bank.domain
org.example.bank.adapter
org.example.bank.application
```

Uygulamaya dışarıdan gelen istekleri temsil eder.

`adapter.in.cli`

Kullanıcı -> CLI / Main -> BankService

CLI, uygulamanın iş mantığını çağırdığı için inbound adapter olarak değerlendirilir.

```text
adapter.in.cli
```

İleride REST API eklendiğinde şu şekilde başka bir inbound adapter oluşturulabilir:

```text
adapter.in.rest
```

`adapter.out.memory`

Uygulamanın dışarıdaki bir saklama sistemine erişmesini temsil eder.

BankService -> Repository interface -> InMemory repository  -> Map veya List

Bu nedenle in-memory repository implementationları şu package altında bulunur:

`adapter.out.memory`

`in.cli` ve `out.memory` birbirinin içinde değildir. İkisi de adapter altında bulunan kardeş package’lardır.

## 5. Repository Interface’leri

Üç farklı repository portu oluşturuldu.

### AccountRepository
```java
public interface AccountRepository {

    Account save(Account account);

    Optional<Account> findById(int accountId);
}
```

Bir hesabı kaydetmek ve ID ile bulmak için kullanılır.

### CustomerRepository
```java
public interface CustomerRepository {

    Customer save(Customer customer);

    Optional<Customer> findById(int customerId);
}
```

Bir müşteriyi kaydetmek ve ID ile bulmak için kullanılır.

### TransactionRepository
```java
public interface TransactionRepository {

    Transaction save(Transaction transaction);

    List<Transaction> findByAccountId(int accountId);
}
```

Bir işlemi kaydetmek ve belirli bir hesabın dahil olduğu işlemleri listelemek için kullanılır.

Transaction repository’de işlem kendi transaction ID’si ile değil, ilgili account ID ile aranmaktadır.

## 6. InMemory Repository Implementationları
InMemoryAccountRepository

Account nesneleri ID ile tekil olarak arandığı için `Map` kullanıldı.

```java
private final Map<Integer, Account> accounts = new HashMap<>();
```
Buradaki yapı:

```text
accountId -> Account
```

şeklindedir.

Kayıt:

```java
accounts.put(account.getAccId(), account);
```

Arama:

```java
Optional.ofNullable(accounts.get(accountId));
InMemoryCustomerRepository
```
Customer nesneleri de Account ile benzer şekilde ID ile tekil olarak arandığı için `Map` kullanıldı.

```java
private final Map<Integer, Customer> customers = new HashMap<>();
```

Kayıt:

```java
customers.put(customer.getAccId(), customer);
```

Arama:

```java
Optional.ofNullable(customers.get(customerId));
InMemoryTransactionRepository
```
Bir hesaba ait birden fazla transaction olabileceği için `List` kullanıldı.
```java
private final List<Transaction> transactions = new ArrayList<>();
```

Transaction kaydetmek için:

```java
transactions.add(transaction);
```

Bir hesabın dahil olduğu işlemleri bulmak için bütün liste dolaşılır.

Transaction şu durumlardan birinde ilgili hesaba aittir:

```java
sourceId == accountId
```

veya:

```java
targetId == accountId
```

Repository yeni bir sonuç listesi döndürür. İçeride tutulan ana transaction listesi doğrudan dışarı verilmez.

## 7. Map ve HashMap

`Map`, anahtar-değer ilişkisiyle veri saklayan bir interface’tir.

```java
Map<Integer, Account>
```

şu anlama gelir:

```text
Integer -> anahtar
Account -> değer
```

Örneğin:

```text
1 -> Account nesnesi
2 -> Account nesnesi
3 -> Account nesnesi
```

`HashMap`, `Map` interface’inin kullanılan somut implementationıdır.

```java
Map<Integer, Account> accounts = new HashMap<>();
```

Burada değişkenin türü interface, oluşturulan nesnenin türü implementationdır.

Temel işlemler:

```java
accounts.put(id, account);
accounts.get(id);
```

Aynı anahtarla tekrar `put()` çağrılırsa eski değer güncellenir.

## 8. Optional

Repository içinde bir nesnenin bulunamama ihtimalini ifade etmek için `Optional` kullanıldı.

```java
Optional<Account> findById(int accountId);
```

Bu method iki farklı sonuç üretebilir:

`Optional` içerisinde `Account`

veya:

```java
Optional.empty()
```

In-memory repository içinde:

```java
return Optional.ofNullable(accounts.get(accountId));
```

kullanıldı.

`Map.get()` sonucu nesne bulunamadığında `null` döndürebilir. `Optional.ofNullable()` bu sonucu güvenli bir `Optional` yapısına dönüştürür.

## 9. BankService Repository Refactoru

BankService içinde daha önce bulunan koleksiyonlar kaldırıldı.

Eski yaklaşım:

```java
private final List<Customer> customerList = new ArrayList<>();
private final List<Account> accountList = new ArrayList<>();
private final List<Transaction> transactionList = new ArrayList<>();
```

Yeni yaklaşım:

```java
private final CustomerRepository customerRepository;
private final AccountRepository accountRepository;
private final TransactionRepository transactionRepository;
```

Böylece BankService artık verilerin hangi koleksiyonda veya hangi teknolojide tutulduğunu bilmemektedir.

Müşteri kaydı

Eski yapı:

```java
customerList.add(customer);
accountList.add(account);
```

Yeni yapı:

```java
customerRepository.save(customer);
accountRepository.save(account);
```
### Login işlemi

Eski yapı bütün müşteri listesini dolaşıyordu.

Yeni yapı:

```java
return customerRepository.findById(userId)
.filter(customer -> customer.passwordMatches(password))
.orElse(null);
```
Repository müşteriyi ID ile bulur, BankService ise şifre kontrolü yapar.

Şifre kontrolü bir iş kuralı olduğu için BankService içinde kalmıştır.

### Account bulma

Tekrarlanan hesap arama işlemi private helper method içinde tutuldu:

```java
private Account findAccountById(int accountId) {
    return accountRepository.findById(accountId)
            .orElseThrow(() ->
                    new IllegalArgumentException(
                            "Hesap bulunamadi: " + accountId
                    )
            );
}
```

### Transaction geçmişi

Eski filtreleme döngüsü BankService içinden kaldırıldı.

Yeni yapı:

```java
public List<Transaction> getTransactionsForAccount(int userId) {
    return transactionRepository.findByAccountId(userId);
}
```

Transaction verisinin nasıl filtreleneceği repository adapterının sorumluluğuna verildi.

## 10. Hesap Güncellemelerinin Kaydedilmesi

Deposit, withdrawal ve transfer işlemlerinde Account nesnesinin bakiyesi değişmektedir.

Başarılı işlemden sonra güncel Account nesnesi repository’ye kaydedilmelidir:

```java
accountRepository.save(account);
```

Transfer işleminde hem gönderen hem alıcı hesap değiştiği için ikisi de kaydedilir:

```java
accountRepository.save(senderAccount);
accountRepository.save(receiverAccount);
```

In-memory repository aynı nesne referansını tuttuğu için şu an tekrar save() yapılmasa da değişiklik görülebilir.

Ancak gelecekte PostgreSQL gibi farklı bir adapter kullanıldığında güncel verinin kalıcı hâle getirilmesi için repository çağrısının açıkça yapılması daha güvenlidir.

## 11. Dependency Injection

Dependency, bir classın çalışmak için ihtiyaç duyduğu başka nesnedir.

BankService’in dependency’leri:

```text
CustomerRepository
AccountRepository
TransactionRepository
```

BankService bu repository’leri kendi içinde oluşturmamaktadır.

Bunun yerine constructor üzerinden dışarıdan almaktadır:

```java
public BankService(
        CustomerRepository customerRepository,
        AccountRepository accountRepository,
        TransactionRepository transactionRepository
) {
    this.customerRepository = customerRepository;
    this.accountRepository = accountRepository;
    this.transactionRepository = transactionRepository;
}
```

Buna constructor dependency injection denir.

BankService şu somut classları bilmez:

```text
InMemoryAccountRepository
InMemoryCustomerRepository
InMemoryTransactionRepository
```

Yalnızca interface’leri bilir.

Bu sayede in-memory implementation yerine gelecekte PostgreSQL implementationı verilebilir.

## 12. Main İçinde Manuel Dependency Injection

Henüz Spring kullanılmadığı için repository nesneleri uygulamanın başlangıç noktasında manuel olarak oluşturulur.

```java
CustomerRepository customerRepository =
        new InMemoryCustomerRepository();

AccountRepository accountRepository =
        new InMemoryAccountRepository();

TransactionRepository transactionRepository =
        new InMemoryTransactionRepository();
```

Daha sonra bu nesneler BankService constructorına verilir:

```java
BankService bankService = new BankService(
        customerRepository,
        accountRepository,
        transactionRepository
);
```

Main şu anda uygulamanın composition root noktasıdır.

Yani uygulamadaki ana nesnelerin oluşturulup birbirine bağlandığı yerdir.

## 13. Unit Testlerde Dependency Injection

BankService’in artık parametresiz constructorı bulunmadığı için testlerde şu kullanım geçerli değildir:

```java
new BankService();
```

Testlerde de BankService’e repository dependency’leri verilmelidir.

Her testten önce temiz repository nesneleri oluşturmak için @BeforeEach kullanılabilir:

private BankService bankService;

```java
@BeforeEach
void setUp() {
    bankService = new BankService(
            new InMemoryCustomerRepository(),
            new InMemoryAccountRepository(),
            new InMemoryTransactionRepository()
    );
}
```

Her test için yeni repository oluşturulması test bağımsızlığını korur.

Bir testte kaydedilen müşteri, hesap veya transaction başka bir testi etkilemez.

## 15. Java İsimlendirme Kuralları

Package isimleri küçük harfle yazılır:

```text
org.example.bank.application.service
org.example.bank.adapter.out.memory
```

Class ve interface isimleri PascalCase kullanır:

```text
BankService
AccountRepository
InMemoryAccountRepository
```

Method ve değişken isimleri lowerCamelCase kullanır:

```text
findById
accountRepository
transactionId
```

Java convention açısından gelecekte şu isimler düzenlenebilir:

```text
userID        -> userId
targetID      -> targetId
transactionID -> transactionId
getAccID()    -> getAccId()
```
