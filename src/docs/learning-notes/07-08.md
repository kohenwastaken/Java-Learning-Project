# Learning Notes

## 2026-07-08 - Service Layer Refactor, Result Enum, Encapsulation ve Transaction Log

Bugün banka CLI projesinde `Main` ve `BankService` arasındaki sorumlulukları daha temiz ayırdık.

### 1. Main ve BankService ayrımı

Önceden birçok iş `Main` içinde yapılıyordu. Bugün bu işleri ikiye ayırdık:

#### Main'in sorumlulukları

- Kullanıcıdan input almak
- Menüleri göstermek
- `Scanner` kullanmak
- Kullanıcıya mesaj yazdırmak
- `BankService` methodlarını çağırmak
- Service'den dönen sonucu kullanıcıya uygun mesajla göstermek

Örnek:

```java
DepositResult result = bankService.depositToAccount(customer.getAccID(), depositAmount);

switch (result){
    case SUCCESS -> IO.println("Guncel bakiye: " + bankService.showBalance(customer.getAccID()));
    case INVALID_AMOUNT -> IO.println("Yatirilan para 0 dan buyuk olmali..");
}
```

#### BankService'in sorumlulukları
- Customer, Account ve Transaction listelerini tutmak
- Yeni müşteri oluşturmak
- Login kontrolü yapmak
- Hesap bulmak
- Para yatırma, çekme, gönderme işlemlerini yönetmek
- Başarılı işlemler için transaction log oluşturmak
- Kullanıcıya ait transactionları filtrelemek

Örnek:
```java
WithdrawResult withdrawFromAccount(int userID, BigDecimal amount) {

    Account account = findAccountByID(userID);

    WithdrawResult result = account.moneyWithdraw(amount);

    if (result == WithdrawResult.SUCCESS) {
        Transaction y = new Transaction(
                this.transactionID++,
                Transaction.TransactionType.WITHDRAWAL,
                amount,
                account.getAccID(),
                null
        );
        this.transactionList.add(y);
    }
    return result;
}
```

Bu ayrım sayesinde `BankService` içinde `IO.println()` veya `Scanner` kalmadı. Bu daha temiz bir service layer yapısıdır.

### 2. Account arama mantığı

Bugün iki farklı hesap arama yöntemi kullandık.

`findAccountByID`

Login olmuş kullanıcının kendi hesabını bulmak için kullanılıyor.

```java
private Account findAccountByID(int userID) {
    for (Account account : this.accountList) {
        if (account.getAccID() == userID) {
            return account;
        }
    }
    throw new IllegalArgumentException("Hesap bulunamadi: " + userID);
}
```

Bu method hesap bulamazsa exception fırlatıyor. Çünkü login olmuş bir kullanıcının hesabının bulunamaması normal kullanıcı hatası değil, sistem tutarsızlığıdır.

`searchAccountByID`

Transferde hedef hesabı bulmak için kullanılıyor.

```java
private Account searchAccountByID(int targetID) {
    for (Account account : this.accountList) {
        if (account.getAccID() == targetID)
            return account;
    }
    return null;
}
```

Bu method hesap bulamazsa `null` döndürüyor. Çünkü transferde hedef ID kullanıcıdan gelir. Kullanıcının yanlış ID girmesi beklenen bir hatadır.

Bu ayrım önemli:
```
Sistemsel tutarsızlık -> exception
Beklenen kullanıcı hatası -> result enum / kontrollü dönüş
```

### 3. Boolean yerine Result Enum kullanımı

Önceden bazı methodlar `boolean` dönüyordu. Fakat özellikle para çekme ve transfer işlemlerinde başarısızlığın birden fazla sebebi var.

Örneğin para çekmede:
```
SUCCESS
INVALID_AMOUNT
INSUFFICIENT_BALANCE
```
Transferde:
```
SUCCESS
ACCOUNT_NOT_FOUND
INVALID_SELF_ID
INVALID_AMOUNT
INSUFFICIENT_BALANCE
```
Bu yüzden ayrı enum dosyaları oluşturduk:

```java
public enum WithdrawResult {
SUCCESS,
INVALID_AMOUNT,
INSUFFICIENT_BALANCE
}
```
```java
public enum DepositResult {
SUCCESS,
INVALID_AMOUNT
}
```
```java
public enum TransferResult{
SUCCESS,
INVALID_AMOUNT,
INVALID_SELF_ID,
INSUFFICIENT_BALANCE,
ACCOUNT_NOT_FOUND
}
```

Böylece `Main` tarafında kullanıcıya daha doğru mesaj gösterebiliyoruz.

Örnek:

```java
switch (result2){
case SUCCESS -> IO.println("Basari ile transfer edildi..");
case ACCOUNT_NOT_FOUND -> IO.println("Hesap bulunamadi..");
case INVALID_SELF_ID -> IO.println("Kendinize para gonderemezsiniz..");
case INVALID_AMOUNT -> IO.println("Gecersiz deger girisi..");
case INSUFFICIENT_BALANCE -> IO.println("Yetersiz bakiye..");
}
```

### 4. Enumları ayrı dosyalara taşıma

Başta enumları `BankService` içinde tutuyorduk. Bu durumda `Account` classı `BankService.WithdrawResult` gibi bir tipe bağımlı oluyordu.

Bu doğru bağımlılık yönü değildi.

Eski bağımlılık:
```
Account -> BankService.WithdrawResult
```
Yeni yapı:
```
Account -> WithdrawResult
BankService -> WithdrawResult
Main -> WithdrawResult
```
Bu daha temizdir. Çünkü `Account`, `BankService` classını bilmek zorunda kalmaz.

### 5. Encapsulation: fieldları private yapmak

Bugün `Account`, `Transaction` ve `Customer` gibi classlarda fieldları `private` yapmanın neden önemli olduğunu konuştuk.

Amaç sadece gizlilik değildir. Asıl amaç, class'ın iç verisinin dışarıdan kontrolsüz değiştirilmesini engellemektir.

Örneğin balance `public` olsaydı başka bir class şunu yapabilirdi:
```java
account.balance = new BigDecimal("-9999");
```

Bu durumda `moneyWithdraw`, `moneyDeposit`, `moneySend` gibi kurallar bypass edilmiş olurdu.

Bu yüzden `Account` içinde:

```java
private final int accID;
private BigDecimal balance;
```

şeklinde bir yapı daha doğrudur.

`accID` değişmemeli, bu yüzden `final` olabilir.
`balance` değişebilir ama sadece kontrollü methodlarla değişmelidir.

Okuma gerekiyorsa getter kullanılır:

```java
public int getAccID() {
return this.accID;
}

public BigDecimal getBalance() {
return this.balance;
}
```

### 7. Transaction log sistemi

Önceden `Main.writeLog()` tüm transaction listesini alıp kendi içinde filtreliyordu.

Bugün bunu değiştirdik.

Artık `BankService` ilgili hesabın transactionlarını filtreliyor:

```java
List<Transaction> getTransactionsForAccount (int userID) {
List<Transaction> result = new ArrayList<Transaction>();
for (Transaction transaction : this.transactionList) {
if (transaction.getSourceID() == userID ||
(transaction.getTargetID() != null) && transaction.getTargetID() == userID) {
result.add(transaction);
}
}
return result;
}
```

`Main` ise sadece gelen listeyi yazdırıyor:

```java
List<Transaction> transactions = bankService.getTransactionsForAccount(customer.getAccID());
writeLog(transactions);
```

Bu ayrım daha doğru oldu:
```
BankService -> hangi transactionlar ilgili?
Main -> bunları nasıl yazdıracağım?
```
Ayrıca boş transaction listesi için mesaj eklendi:

```java
if (transactionList.isEmpty()) {
IO.println("Henuz islem gecmisi yok.");
return;
}
```

### 8. Bugünkü önemli prensipler

Bugün özellikle şu prensipler oturdu:

```
Main input/output ile ilgilenir.
BankService business flow ile ilgilenir.
Account kendi balance kurallarını korur.
Transaction kayıtları başarılı işlemlerde oluşturulur.
Fieldlar mümkün olduğunca private olmalıdır.
ID gibi değişmemesi gereken değerler final olabilir.
Boolean yerine gerektiğinde enum result kullanılabilir.
Service iç listeleri dışarıya doğrudan açılmaz.
```
