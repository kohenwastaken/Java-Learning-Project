# İşlem Geçmişi ve Postman Düzeni

## Günün Amacı

Bu çalışmada hesap işlem geçmişi REST API’ye bağlandı ve Postman istekleri daha düzenli hâle getirildi.

Tamamlanan başlıca konular:

- İşlem geçmişi endpoint’i
- `TransactionResponse` DTO’su
- Domain listesinin response listesine dönüştürülmesi
- Boş liste davranışı
- Postman collection ve variable kullanımı
- Rastgele test verileri
- Postman ile REST API arasındaki fark

## İşlem Geçmişi Endpoint’i

Endpoint:

```text
GET /api/accounts/{accountId}/transactions
```

Örnek istek:

```text
GET /api/accounts/1/transactions
```

Bu endpoint herhangi bir yeni işlem oluşturmaz. Mevcut işlemleri okuduğu için `GET` kullanır ve request body içermez.

## `TransactionResponse` DTO’su

Domain katmanındaki `Transaction` nesnesini doğrudan dışarıya açmak yerine ayrı bir response DTO oluşturuldu.

```java
public record TransactionResponse(
        int transactionId,
        Transaction.TransactionType type,
        BigDecimal amount,
        int sourceId,
        Integer targetId
) {
}
```

`targetId` için `Integer` kullanılmasının nedeni bazı işlemlerde hedef hesabın bulunmamasıdır.

Para yatırma ve para çekme işlemlerinde:

```json
"targetId": null
```

dönebilir.

Transfer işleminde ise hedef hesap ID’si bulunur.

## Domain Listesini Response Listesine Dönüştürme

`BankService.getTransactionsForAccount(accountId)` methodu şu türde veri döndürür:

```java
List<Transaction>
```

Controller bu domain nesnelerini dışarıya uygun DTO’lara dönüştürür:

```java
List<TransactionResponse>
```

Temel dönüşüm akışı:

```text
List<Transaction>
        ↓
Her Transaction için TransactionResponse oluştur
        ↓
List<TransactionResponse>
        ↓
Spring tarafından JSON dizisine dönüştürülür
```

Örnek response:

```json
[
  {
    "transactionId": 1,
    "type": "DEPOSIT",
    "amount": 500,
    "sourceId": 1,
    "targetId": null
  },
  {
    "transactionId": 2,
    "type": "WITHDRAWAL",
    "amount": 200,
    "sourceId": 1,
    "targetId": null
  },
  {
    "transactionId": 3,
    "type": "TRANSFER",
    "amount": 100,
    "sourceId": 1,
    "targetId": 2
  }
]
```

## Boş Liste Davranışı

Hesapta hiç işlem yoksa endpoint şu cevabı döndürür:

```json
[]
```

Bu doğru bir API davranışıdır. Anlamı:

```text
İstek başarılı oldu fakat gösterilecek işlem bulunmuyor.
```

`null` dönmek yerine boş liste dönmek istemci tarafında daha güvenli ve anlaşılırdır.

Mevcut yapıda var olmayan hesap da boş liste döndürebilir. İleride hata yönetimi eklenerek:

```text
Mevcut hesap + işlem yok → 200 OK ve []
Var olmayan hesap       → 404 Not Found
```

ayrımı yapılacaktır.

## Postman Collection

Bütün bankacılık istekleri `Bank API` adlı bir collection altında kaydedildi.

Kaydedilen temel request’ler:

```text
Register Customer
Get Balance
Deposit Money
Withdraw Money
Transfer Money
Get Transactions
```

Her request’in HTTP methodu, URL’si ve JSON body’si kaydedildiği için aynı alanları tekrar tekrar yazmak gerekmemektedir.

## Collection Variable

Ortak sunucu adresi collection variable olarak tanımlandı:

```text
baseUrl = http://localhost:8080
```

Böylece URL’ler şu biçimde kullanılabilir:

```text
{{baseUrl}}/api/customers
{{baseUrl}}/api/accounts/1/balance
{{baseUrl}}/api/accounts/1/deposits
```

Sunucu adresi değişirse bütün request’leri ayrı ayrı düzenlemek yerine yalnızca `baseUrl` değiştirilir.

## Rastgele Test Verileri

Müşteri kayıtlarında aynı isimleri sürekli elle değiştirmemek için Postman’ın dinamik değişkenleri kullanıldı.

```json
{
  "name": "{{$randomFirstName}}",
  "surname": "{{$randomLastName}}",
  "password": "1234"
}
```

Her istek gönderildiğinde Postman farklı örnek ad ve soyad üretir. Böylece test verileri daha kolay çeşitlendirilir.

Hesap ID’lerini otomatik saklayan script yapısı incelendi ancak mevcut proje için gerekli görülmedi. Basit ve anlaşılır kullanım tercih edildi.

## Postman REST API Değildir

Postman, REST API’ye istek gönderen bir araçtır.

```text
Postman
   ↓ HTTP request
Spring Boot REST API
   ↓
Controller
   ↓
BankService
   ↓
Repository
```

REST API, Spring Boot uygulamasında oluşturulan endpoint’lerin bütünüdür.

Postman ise bu endpoint’leri geliştirme sırasında çağırmak, test etmek ve cevapları incelemek için kullanılır.

Gerçek bir web frontend’i veya mobil uygulama da ileride aynı endpoint’lere HTTP isteği gönderebilir.

## REST Tarafında Tamamlanan Endpoint’ler

```text
POST /api/customers
GET  /api/accounts/{accountId}/balance
POST /api/accounts/{accountId}/deposits
POST /api/accounts/{accountId}/withdrawals
POST /api/accounts/{accountId}/transfers
GET  /api/accounts/{accountId}/transactions
```

Bu aşamada temel bankacılık özelliklerinin tamamı Spring Boot REST API üzerinden kullanılabilir ve Postman ile test edilebilir durumdadır.

## Sonraki Aşama

REST API’nin temel işlevleri tamamlandıktan sonra sıradaki büyük aşama PostgreSQL kalıcılığıdır.

Mevcut yapı:

```text
BankService
→ Repository interface
→ InMemory repository
→ Map / List
```

Hedef yapı:

```text
BankService
→ Aynı repository portları
→ PostgreSQL adapter
→ PostgreSQL
```

Hexagonal mimari sayesinde `BankService` veritabanı teknolojisine doğrudan bağımlı değildir.
