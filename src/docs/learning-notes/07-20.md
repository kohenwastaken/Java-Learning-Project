## 1. Endpoint Nedir?

Endpoint, uygulamanın dışarıdan erişilebilen belirli bir işlemini temsil eden HTTP adresi ve yöntemidir.

Örneğin:

```text
GET /api/hello
```

ve:

```text
POST /api/customers
```

iki farklı endpoint’tir.

Endpoint yalnızca URL’den oluşmaz. HTTP yöntemi de endpoint’in bir parçasıdır.

```text
GET  /api/customers
POST /api/customers
```

Adresleri aynı olsa bile farklı işlemleri temsil edebilirler.

Endpoint, uygulamadaki bir işlemin dış dünyaya açılan kapısı olarak düşünülebilir.

---

## 2. GET ve POST Yöntemleri

### GET

GET, çoğunlukla bir veriyi okumak veya görüntülemek için kullanılır.

Örnek:

```text
GET /api/hello
```

Bu isteğin amacı uygulamadan bir cevap almaktır.

Tarayıcının adres çubuğuna bir URL yazıldığında tarayıcı genellikle GET isteği gönderir.

### POST

POST, uygulamaya veri göndermek ve çoğunlukla yeni bir kayıt veya işlem oluşturmak için kullanılır.

Örnek:

```text
POST /api/customers
```

Bu istekte müşteri bilgileri JSON gövdesi içerisinde gönderilir:

```json
{
  "name": "Ahmet",
  "surname": "Yılmaz",
  "password": "1234"
}
```

GET veya POST isteğini annotationlar oluşturmaz. İstek dışarıdaki bir istemci tarafından gönderilir.

İstek gönderebilecek istemciler:

```text
Tarayıcı
Postman
Mobil uygulama
Frontend uygulaması
Başka bir sunucu
curl
```

Spring, dışarıdan gelen isteği uygun controller methoduyla eşleştirir.

## 3. DTO Nedir?

DTO, `Data Transfer Object` ifadesinin kısaltmasıdır.

Türkçe karşılığı veri taşıma nesnesidir.

DTO’nun temel görevi, uygulamanın dışından gelen veya dışına gönderilen verileri taşımaktır.

Bugünkü müşteri kayıt işleminde iki DTO kullanıldı:

```text
RegisterCustomerRequest
CustomerResponse
```

### RegisterCustomerRequest

HTTP isteğinden gelen müşteri bilgilerini taşır:

```text
name
surname
password
```

### CustomerResponse

HTTP cevabında dışarıya gösterilecek bilgileri taşır:

```text
accountId
name
surname
```

Response içerisinde parola alanı bulunmaz. Böylece kullanıcı parolası HTTP cevabında dışarıya gönderilmez.

DTO bir Spring özelliği değildir. Bir tasarım yaklaşımıdır ve normal Java classı veya record kullanılarak oluşturulabilir.

## 4. Neden Domain Nesnesi Doğrudan Kullanılmadı?

Mevcut domain nesnesi:

```text
Customer
```

uygulamanın iç işleyişini temsil eder.

Ancak müşteri kayıt isteğinde kullanıcıdan account ID alınmamalıdır. ID sistem tarafından oluşturulmalıdır.

Ayrıca cevap verirken parola gönderilmemelidir.

Bu nedenle HTTP sınırında ayrı nesneler kullanıldı:

```text
Gelen JSON
    ↓
RegisterCustomerRequest
    ↓
CustomerController
    ↓
BankService
    ↓
Customer
    ↓
CustomerResponse
    ↓
Giden JSON
```

Bu yapı dış dünya ile domain modelinin doğrudan birbirine bağlanmasını engeller.

## 5. Java `record` Yapısı

`record`, veri taşımaya yönelik basit Java classlarını kısa şekilde tanımlamak için kullanılır.

Normal bir class içerisinde şu parçaların yazılması gerekebilir:

- Fieldlar
- Constructor
- Getterlar
- `equals()`
- `hashCode()`
- `toString()`

Record bunların büyük kısmını otomatik oluşturur.

Örnek:

```java
public record RegisterCustomerRequest(
        String name,
        String surname,
        String password
) {
}
```

Record nesnesi normal bir nesne gibi oluşturulur:

```java
RegisterCustomerRequest request =
        new RegisterCustomerRequest(
                "Ahmet",
                "Yılmaz",
                "1234"
        );
```

Record içerisindeki verilere getter biçiminde değil, bileşen adıyla erişilir:

```java
request.name();
request.surname();
request.password();
```

Normal classlarda kullanılan:

```java
request.getName();
```

yerine record içerisinde:

```java
request.name();
```

kullanılır.

Record bileşenleri oluşturulduktan sonra yeniden atanamaz. Bu nedenle request ve response DTO’ları gibi veri taşıma amaçlı nesnelerde uygundur.

## 6. Kullanılan Spring Annotationları

### `@RestController`

```java
@RestController
```

Bir classın HTTP isteklerini karşılayan controller methodları içerdiğini Spring’e bildirir.

Controller içindeki methodların dönüş değerleri HTTP cevabı olarak gönderilebilir.

---

### `@RequestMapping`

```java
@RequestMapping("/api/customers")
```

Controller içerisindeki endpoint’ler için ortak bir başlangıç adresi tanımlar.

Tek başına hangi methodun çalışacağını veya isteğin GET ya da POST olduğunu belirlemez.

Örnek:

```java
@RestController
@RequestMapping("/api/customers")
public class CustomerController {
}
```

Bu controller içerisindeki endpoint’lerin adresleri `/api/customers` ile başlar.

---

### `@GetMapping`

```java
@GetMapping("/hello")
```

Belirtilen adrese GET isteği geldiğinde ilgili Java methodunun çalıştırılmasını sağlar.

Örnek:

```java
@GetMapping("/hello")
public String hello() {
    return "Banka uygulamasi calisiyor.";
}
```

Controller üzerinde:

```java
@RequestMapping("/api")
```

bulunduğunda tam endpoint:

```text
GET /api/hello
```

olur.

---

### `@PostMapping`

```java
@PostMapping
```

Belirtilen adrese POST isteği geldiğinde ilgili Java methodunun çalıştırılmasını sağlar.

Controller üzerinde:

```java
@RequestMapping("/api/customers")
```

ve method üzerinde:

```java
@PostMapping
```

bulunduğunda tam endpoint:

```text
POST /api/customers
```

olur.

---

### `@RequestBody`

```java
@RequestBody RegisterCustomerRequest request
```

HTTP isteğinin gövdesindeki JSON verisini okuyup belirtilen Java nesnesine dönüştürür.

Gelen JSON:

```json
{
  "name": "Ahmet",
  "surname": "Yılmaz",
  "password": "1234"
}
```

Spring tarafından şu nesneye dönüştürülür:

```java
RegisterCustomerRequest request
```

Daha sonra verilere şu şekilde erişilir:

```java
request.name();
request.surname();
request.password();
```

## 7. CustomerController

İlk gerçek sistem endpoint’i için `CustomerController` oluşturuldu.

Temel yapı:

```java
@RestController
@RequestMapping("/api/customers")
public class CustomerController {

    private final BankService bankService;

    public CustomerController(BankService bankService) {
        this.bankService = bankService;
    }

    @PostMapping
    public CustomerResponse registerCustomer(
            @RequestBody RegisterCustomerRequest request
    ) {
        Customer customer = bankService.registerCustomer(
                request.name(),
                request.surname(),
                request.password()
        );

        return new CustomerResponse(
                customer.getAccId(),
                customer.getName(),
                customer.getSurname()
        );
    }
}
```

Controller’ın sorumlulukları:

1. HTTP isteğini almak
2. Gelen JSON’u request DTO üzerinden okumak
3. BankService’i çağırmak
4. Domain sonucunu response DTO’ya çevirmek
5. Sonucu Spring’e döndürmek

Controller müşteri kayıt iş mantığını kendisi yürütmez.

Gerçek kayıt işlemi hâlâ:

```java
bankService.registerCustomer(...)
```

methodu tarafından gerçekleştirilir.

## 8. Controller Dependency Injection

CustomerController, BankService nesnesini kendi içerisinde oluşturmaz.

Yanlış yaklaşım:

```java
private final BankService bankService = new BankService(...);
```

Kullanılan yaklaşım:

```java
private final BankService bankService;

public CustomerController(BankService bankService) {
    this.bankService = bankService;
}
```

Spring, daha önce bean olarak tanımlanmış BankService nesnesini controller constructorına verir.

Akış:

```text
Spring BankService bean’ini oluşturur.
            ↓
CustomerController oluşturulur.
            ↓
BankService constructor üzerinden controller’a verilir.
```

## 9. İlk Gerçek REST Akışı

Postman’dan şu istek gönderildi:

```text
POST http://localhost:8080/api/customers
```

Request body:

```json
{
  "name": "Ahmet",
  "surname": "Yılmaz",
  "password": "1234"
}
```

Çalışma akışı:

```text
Postman
    ↓
POST /api/customers
    ↓
Spring Boot
    ↓
CustomerController
    ↓
RegisterCustomerRequest
    ↓
BankService.registerCustomer()
    ↓
Customer ve Account oluşturulması
    ↓
In-memory repository’lere kaydetme
    ↓
CustomerResponse
    ↓
JSON cevap
```

Dönen cevap:

```json
{
  "accountId": 1,
  "name": "Ahmet",
  "surname": "Yılmaz"
}
```

HTTP durum kodu:

```text
200 OK
```

olarak gözlemlendi.

## 10. Tarayıcıdan Alınan 400 Bad Request Hatası

Müşteri endpoint’i doğrudan tarayıcıdan açıldığında:

```text
http://localhost:8080/api/customers
```

hata ekranı görüldü.

Bunun nedeni tarayıcının adres çubuğundan otomatik olarak GET isteği göndermesidir.

Ancak oluşturulan endpoint:

```text
POST /api/customers
```

isteğini ve JSON gövdesini beklemektedir.

Tarayıcıdan gönderilen istekte gerekli JSON body bulunmadığı için uygun işlem gerçekleştirilemedi.

Bu endpoint Postman üzerinden:

- POST yöntemi
- Doğru URL
- JSON body
- `Content-Type: application/json`

ile test edildiğinde başarılı şekilde çalıştı.

## 11. Postman Nedir?

Postman, API endpoint’lerine HTTP istekleri göndermek ve gelen cevapları incelemek için kullanılan ayrı bir uygulamadır.

Spring Boot ile farkı:

```text
Spring Boot
→ HTTP isteğini karşılayan Java uygulamasıdır.

Postman
→ HTTP isteğini hazırlayıp gönderen test aracıdır.
```

Postman Homebrew üzerinden kuruldu:

```bash
brew install --cask postman
```

Postman masaüstü uygulaması kullanıldığı için localhost üzerindeki Spring Boot uygulamasına doğrudan istek gönderilebildi.

## 13. CLI ve REST Arasındaki Karşılıklar

CLI sisteminde Main içerisinde kullanılan yapı:

```java
while (true) {
    switch (choice) {
        // işlemler
    }
}
```

REST tarafında doğrudan yazılmadı.

Bunun nedeni Spring Boot web sunucusunun sürekli HTTP isteği beklemesidir.

### CLI karşılıkları

```text
while
→ Programı açık tutar ve kullanıcıdan sürekli seçim bekler.

switch
→ Kullanıcı seçimine göre çalıştırılacak işlemi belirler.

Scanner
→ Kullanıcıdan veri alır.

IO.println
→ Sonucu terminale yazar.
```

### REST karşılıkları

```text
Spring web sunucusu
→ Sürekli HTTP isteği bekler.

@GetMapping / @PostMapping
→ Gelen isteğe göre çalıştırılacak methodu belirler.

@RequestBody
→ Gönderilen JSON verisini alır.

Response DTO
→ Sonucu JSON olarak istemciye gönderir.
```

Karşılaştırma:

| CLI Main | REST API |
|---|---|
| `while` | Spring web sunucusunun istek beklemesi |
| `switch` | URL ve HTTP method eşleştirmesi |
| `Scanner` | JSON ve `@RequestBody` |
| `IO.println` | Response DTO ve JSON |
| Menü seçimi | Endpoint’e istek gönderme |

BankService her iki yapıda da aynı iş mantığını yürütmeye devam eder.
